## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client browser -> Server Action (POST) | Server Actions are directly POST-reachable; must re-verify auth, not trust the rendering page |
| client browser -> PostgREST | The edit write still lands under the Plan 01 column grant + RLS — app validation is defense-in-depth, DB is load-bearing |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-05 | Spoofing | forged/stale cookie presented to the Server Action | mitigate | `updateProfileAction` re-verifies via `getUser()` (never getSession) inside the action |
| T-04-06 | Elevation of Privilege | edit payload smuggles `level`/`role` | mitigate | zod schema has no `level`/`role` key; updateSafeFields type excludes `level`; DB grant/trigger (Plan 01) is the load-bearing backstop |
| T-04-07 | Denial of Service (minor) | oversized `goal` text | mitigate | zod `.max(2000)` on goal before the DB write |
| T-04-08 | Tampering | preference CSS applied via inline style (silent no-op / FOUC exploit) | accept-then-mitigate | Author theme/text-size/reduced-motion as stylesheet rules only (Pitfall 5); JS flips data-* attributes exclusively |
</threat_model>
