<objective>
Create the phase's real database in a strictly correct migration order: a hardened `profiles` table (row auto-created on signup by a trigger that never silently blocks signup), a `coach_clients` join table with a one-coach-per-client UNIQUE constraint AND role-integrity enforcement, RLS on every table routed through a `SECURITY DEFINER` helper that is defined only after `coach_clients` exists (no forward reference), and a role self-escalation guard. Push the migrations to local Supabase, verify the trigger/RLS/escalation guards against a real instance, and regenerate types into a dedicated generated file so hand-written legacy chat contracts never drift.

Purpose: DB-01..04. Research is explicit — every DB-* requirement lives entirely in the Database tier and must be provably correct there (RLS tested per role, trigger tested against malformed metadata) before any auth UI is built on top. The regenerated types are what the auth screens in wave 3 import via `@fish/supabase`.

Output: Five ordered migrations, the applied schema on local Supabase, verified guards, and a `database.generated.ts` + a composed `database.types.ts`.

Review-driven changes in this revision (from 02-REVIEWS.md):
- HIGH: `private.is_coach_of()` moved OUT of the profiles migration into `0004_rls_helpers.sql`, which runs AFTER `0003_coach_clients.sql` creates the table it references — `supabase db reset` can no longer fail on a forward-referenced relation.
- MEDIUM: `profiles` now has an explicit authenticated UPDATE policy scoped to safe fields (display_name) so the role-escalation guard is genuinely exercised (previously an absent UPDATE policy could block the test at the RLS layer before it ever reached the trigger).
- MEDIUM: `coach_clients` gains role-integrity enforcement (coach_id must be a coach profile, client_id a client profile) AND `is_coach_of()` additionally verifies the caller's own role is coach — a malformed row can no longer leak assigned profiles.
- MEDIUM: generated types split into `database.generated.ts`; `database.types.ts` composes them and keeps the legacy `conversations`/`messages` chat contracts as clearly-marked not-yet-migrated placeholders instead of mixing generated + hand-written entries in one regenerated file.
</objective>
