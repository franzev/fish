<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| auth.users insert → profiles | Trigger runs SECURITY DEFINER in the same transaction as signup |
| authenticated client → profiles.role | The most sensitive column; self-writes must be blocked |
| RLS policy → protected table | A policy that queries its own table recurses (42P17) |
| service_role → schema | The seed script bypasses RLS by design |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-04 | Elevation of Privilege | profiles.role self-update | mitigate | `prevent_role_self_escalation` trigger blocks authenticated role changes; `handle_new_user` hard-codes `'client'` (defense in depth) — verified positively in Task 3 |
| T-02-05 | Denial of Service | RLS policy recursion (42P17) | mitigate | All coach/client lookups route through `private.is_coach_of()` SECURITY DEFINER STABLE helper; no bare self-SELECT in any policy (Task 1/2 grep gates) |
| T-02-06 | Denial of Service | handle_new_user aborts signup transaction | mitigate | `security definer` + `set search_path = ''` + `on conflict do nothing` + `coalesce` + NOT NULL DEFAULT role — prevention not recovery (Pitfall 4); verified with malformed-metadata signup in Task 3 |
| T-02-07 | Elevation of Privilege | role read from user_metadata | mitigate | Role lives only in `profiles.role`; no policy or check reads `raw_user_meta_data` for authorization (Pitfall 3) |
| T-02-08 | Information Disclosure | client reads another user's data | mitigate | RLS SELECT policies scope reads to own row / assigned clients; verified per-role in Task 3 (DB-03) |
</threat_model>
