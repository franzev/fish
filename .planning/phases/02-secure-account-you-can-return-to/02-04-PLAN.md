<objective>

</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser form → Supabase Auth | Untrusted signup input crosses into GoTrue |
| email link → /auth/confirm | A single-use token_hash arrives via an unauthenticated GET (scanners may pre-fetch) |
| /home server read | Session must be verified server-side, not trusted from the cookie |
| signup metadata → role | A client could try to pass role='coach' in signUp metadata |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-13 | Elevation of Privilege | signUp metadata role injection | mitigate | Role never read from signup metadata; `handle_new_user` hard-codes `'client'` (plan 02); signup only sends `display_name` (Task 1) |
| T-02-14 | Spoofing | /home trusting a forged cookie | mitigate | `/home` reads `getUser()` server-side, never `getSession()` (Task 3 grep gate, Pitfall 1) |
| T-02-15 | Denial of Service | email scanner burning single-use token | mitigate | Consumed/expired token → calm `/expired-link` with resend, never a raw error (D-06, Task 2) |
| T-02-16 | Information Disclosure | verification email leaking tokens in URL fragment | mitigate | Template uses `token_hash` query param + `/auth/confirm`, never `{{ .ConfirmationURL }}` implicit-flow fragment (Pitfall 6, Task 3) |
| T-02-17 | Session fixation / dropped refresh | session lost on restart | mitigate | proxy.ts refresh (plan 01) writes cookies to request+response; verified across restart in Task 4 (AUTH-05) |
</threat_model>
