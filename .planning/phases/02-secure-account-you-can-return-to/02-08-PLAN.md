<objective>
Close two related UAT gaps left after phase 02's earlier gap-closure plans:

1. UAT test 11 (major): /expired-link never submits on Enter. The email Input and resend Button sit in a bare `<div>` with `type="button"` + `onClick`, so implicit form submission (Enter in a text field) is a structural no-op — it is the ONLY auth screen deviating from the sibling `<form onSubmit>` + `type="submit"` pattern. /check-inbox has the identical no-form shape; the defect is latent there only because that screen has no text field to press Enter in.

2. Cursor feedback (cosmetic, pre-diagnosed): Button sets no cursor utilities, so Tailwind v4 preflight leaves `<button>` at `cursor:default` and there is no pointer feedback. Compounding: `disabled:pointer-events-none` and the loading `pointer-events-none` suppress the element's OWN cursor, so `cursor-progress`/`cursor-not-allowed` can never take effect while those classes remain.

UAT test 10 (/reset-password Enter-submit) is intentionally NOT a code task here: the diagnosis (`.planning/debug/enter-submit-reset-password.md`) proved that page's `<form onSubmit>` + `type="submit"` submits on Enter end-to-end when hydrated; the report matched the next-dev pre-hydration window on the first route visit. It closes via a UAT retest on a settled/hydrated page, captured in the human checkpoint below — no /reset-password code change, and no jsdom test (jsdom cannot exercise native implicit submission).

Purpose: Make Enter submit on every auth screen (bring the two stragglers onto the shared form pattern) and give Button honest cursor affordances while keeping disabled/loading genuinely non-activatable and the size perfectly stable.

Output: /expired-link and /check-inbox each submit their resend through a real `<form onSubmit>` with a `type="submit"` Button; Button renders cursor-pointer / cursor-progress / cursor-not-allowed with the disabled/loading states still inert to click and keyboard.
</objective>
