## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| seed script → Supabase admin API | The service-role key grants full bypass of RLS; must never leak to the client |
| seed → auth.users | Seeded users must pass through GoTrue, not raw SQL, to fire the real trigger |
| service_role → profiles.role | The one legitimate path that may set role='coach' |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-09 | Elevation of Privilege | seed role promotion | mitigate | Coach role set only via service_role update (bypasses the escalation guard by design); the guard still blocks authenticated callers (verified Task 2, Assumption A3) |
| T-02-10 | Information Disclosure | service-role key in seed | mitigate | Key read from `SUPABASE_SERVICE_ROLE_KEY` (never `NEXT_PUBLIC_`), env confined to the seed's runtime; `.env.local` git-ignored (plan 01) |
| T-02-11 | Tampering | raw auth.users insert bypassing GoTrue | mitigate | Seed uses `admin.createUser` only; grep gate forbids `insert into auth.users` (Task 1) |
| T-02-12 | Information Disclosure | dev credentials reaching production | accept | Dev credentials are fixed and documented for local only; deploy checklist explicitly bars running the seed in prod (Task 3) |
</threat_model>
