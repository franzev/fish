<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| login form → Supabase Auth | Untrusted credentials cross into GoTrue |
| forgot-password response → user | Response must not reveal whether an account exists |
| reset link → recovery session | A single-use recovery token_hash establishes a password-change session; the destination is fixed by the template's next |
| reset-password → updateUser | Only the holder of the recovery session may set a new password |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-18 | Information Disclosure | account enumeration via forgot-password | mitigate | Identical success copy for existing + non-existent emails; no UI branch on account existence (D-07, Task 2) |
| T-02-19 | Information Disclosure | login error reveals which field is wrong | mitigate | Single combined "email and password don't match" error; never field-specific (Task 1) |
| T-02-20 | Elevation of Privilege | password change without a valid recovery session | mitigate | reset-password relies on the recovery session issued server-side by `/auth/confirm` (type=recovery); `updateUser` fails without it (Task 2, D-08) |
| T-02-21 | Information Disclosure | reset email leaking token in URL fragment | mitigate | recovery.html uses `token_hash` query param + `/auth/confirm`, never `{{ .ConfirmationURL }}` (Pitfall 6, Task 2) |
| T-02-22 | Denial of Service | consumed reset link → raw error | mitigate | Expired/used recovery token → calm `/expired-link` (reuses plan 04, D-06); verified Task 3 |
| T-02-28 | Denial of Service | recovery link drops/mangles next, dead-ends the reset | mitigate | Template hardcodes `next=/reset-password` (simple path, no encoding); the Mailpit URL is read and confirmed in Task 3 before relying on it (review HIGH) |
</threat_model>
