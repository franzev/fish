## Spacing Scale

Unchanged from Phase 1 — no new spacing tokens needed. Auth screens are single-column forms that reuse the existing 4px-multiple rhythm already present in `Input`/`Button`/`Card`.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-text gaps (Input notice/error icon spacing — existing) |
| sm | 8px | Compact spacing — label-to-input gap, hint/notice text margin-top |
| md | 16px | Default element spacing — form field stacking inside auth Cards |
| lg | 24px | Card padding (existing `p-6`), form-to-link spacing below the primary action |
| xl | 32px | Card-to-viewport-edge breathing room on the centered auth layout |
| 2xl | 48px | Vertical centering offset / top padding on auth pages |
| 3xl | 64px | Not used this phase (no long scrolling page like `/kit`) |

Exceptions:
- Every control (`Input`, `Button`) stays at the existing 56px floor (`--size-control`, KIT-04) — no exception for this phase's forms, including the single-field reset-password screen.
- No new spacing tokens introduced. Auth pages reuse the same `max-w-[440px]` centered-column pattern already established on `/kit` (`apps/web/app/kit/page.tsx`), applied to a vertically centered `Card` instead of a long scroll.

## Typography

Unchanged from Phase 1 — no new sizes or weights. Auth screens use the existing 4-size, 2-weight scale exactly as declared in `01-UI-SPEC.md`.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 17px | 400 (regular) | 1.55 |
| Label / Caption | 14px | 400 (regular) / 500 (medium, field labels only) | 1.4 |
| Heading (Card title, e.g. "Log in") | 20px | 600 (semibold) | 1.2 |
| Display (page-level, used sparingly — this phase's Cards use Heading, not Display) | 32px | 600 (semibold) | 1.15 |

Weights: exactly 2 — 400 and 600, same as Phase 1. Auth screen headings (D-03: "Fraunces heading") use the 20px Heading role, not the 32px Display role — auth Cards are compact, not hero moments; Display stays reserved for the `/kit` page title precedent and any future marketing/landing use.

## Color

Unchanged from Phase 1's monochrome `light-dark()` ladder in `apps/web/app/globals.css`. This phase introduces zero new color tokens — every auth screen state (default, hover, focus, disabled, notice, error) is expressed with the existing tokens below.

| Role | Usage in this phase |
|------|------|
| `bg` (60% dominant) | Auth page canvas behind the centered Card |
| `surface` / `surface-2` (30% secondary) | Card background (`surface`), any inset/well |
| `border` / `border-strong` | Input default border / notice-tier border |
| `primary` (10% accent) | The ONE primary Button per auth screen — Sign up, Log in, Send reset link, Set new password, Resend email, Log out |
| `on-primary` | Text on the primary button fill |
| `foreground` / `body` / `muted` | Card heading / body copy / hint text and the quiet sibling-flow link |
| `notice` | Field-level guidance that isn't yet an error (e.g. password hint, "at least 8 characters") |
| `error` | Field-level and form-level (`Alert`) validation failures — wrong password, weak password, expired link |
| `success` | Not used as a persisted screen state this phase (no green checkmarks) — the closest analog, the "check your inbox" and "reset link sent" screens, use `Alert tone="notice"` or a plain Card body, not `tone="success"`, because they are process confirmations, not completed achievements |

Accent (`primary`) reserved for, this phase specifically:
- Sign up (signup screen)
- Log in (login screen)
- Send reset link (forgot-password screen)
- Set new password (reset-password screen)
- Resend the email (check-inbox / expired-link screens)
- Log out (`/home` screen)

Exactly one `primary` Button per screen at all times (AGENTS.md rule 1) — every auth screen in this phase has exactly one. The "sibling flow" link (e.g. "New here? Create account" on `/login`) is plain text, never a `ghost`/`secondary` Button, so it never reads as a second competing action (D-03).

Second semantic color (destructive): this phase's only destructive-flavored action is **Log out**, and per AGENTS.md/Phase 1 precedent there is no red — log out is a plain, unscary primary action (D-01: "a calm 'you're signed in' confirmation plus the log out action"), not styled as destructive at all. No `error`-tinted button exists in this phase. Field/form validation failures (wrong password, expired link, weak password) use the existing `error` token exactly as Phase 1 defined it — semibold weight, heavier border, monochrome, never red.

No new tokens, no hue, nothing added to `globals.css` by this phase.

## Copywriting Contract

Drafted in FISH voice per CONTEXT.md's discretion note ("Exact screen and email copy — drafted in FISH voice... reviewed at phase verification"). Sentence case, plain verbs, never scolds, never reveals account existence (D-07), routes rather than errors where possible (D-05/D-06).

| Element | Copy |
|---------|------|
| Primary CTA — Signup | "Create account" |
| Primary CTA — Login | "Log in" |
| Primary CTA — Forgot password | "Send reset link" |
| Primary CTA — Reset password (set new password) | "Set new password" |
| Primary CTA — Check-inbox / expired-link resend | "Resend the email" |
| Primary CTA — Home (logout) | "Log out" |
| Signup heading | "Create your account" |
| Signup password hint | "At least 8 characters." (D-16 — stated upfront, not after a failed attempt) |
| Signup sibling link | "Already have an account? Log in" |
| Login heading | "Log in" |
| Login sibling link | "New here? Create account" |
| Login sibling link (forgot password) | "Forgot your password?" |
| Check-inbox heading | "Check your inbox" |
| Check-inbox body | "We sent a link to {email}. Open it on this device to continue." |
| Check-inbox resend confirmation (after tapping Resend) | "Sent again. Check your inbox." (Alert `tone="notice"`, not a page change) |
| Expired-link heading | "That link has expired" |
| Expired-link body | "Links only work once, and this one's had its turn. Send yourself a fresh one." |
| Forgot-password heading | "Reset your password" |
| Forgot-password body | "Enter the email on your account and we'll send you a reset link." |
| Forgot-password success (D-07 — identical regardless of account existence) | "If that address has an account, a reset link is on its way." (shown in place of the form after submit, not as a separate route — no account-enumeration, no dead end) |
| Reset-password heading | "Set a new password" |
| Reset-password body | Single field only: new password, hint "At least 8 characters." |
| Home heading | "You're signed in" |
| Home body | "This confirms your session — nothing else lives here yet." |
| Empty state heading | Not applicable this phase — no data-bearing empty states exist yet (Phase 3 scope, SHEL-02). Tone rule carried forward: never "No data found." |
| Error state — wrong email/password on login | Field-level `Input error`: "That email and password don't match. Try again?" (never "Invalid credentials," never reveals which field is wrong) |
| Error state — unverified login attempt | Not a field error — routes to the check-inbox screen per D-05 ("login never scolds, it just routes") |
| Error state — signup with existing email | Field-level `Input error` on email: "That email's already in use. Try logging in instead?" (guides to the fix, doesn't just refuse) |
| Error state — password too short | Field-level `Input error` on password: "Needs to be at least 8 characters." |
| Error state — generic network/server failure | Form-level `Alert tone="error"`: "Something needs your attention before you can continue. Try again in a moment." (existing Phase 1 example copy, reused verbatim — no new error language invented) |
| Destructive confirmation — Log out | No confirmation dialog. Per AGENTS.md "gamification is reward-only" spirit and the calm/frictionless product stance, logging out is a single click of the one `primary` Button on `/home` — sessions are cheap to re-establish (AUTH-05 already proves persistence), so a confirm-dialog would add a choice/step the product philosophy explicitly avoids. This is a deliberate exception to a generic "confirm destructive actions" rule: logout is reversible in one login, not data-destroying. |

Copy rules inherited from AGENTS.md and Phase 1 (binding, restated for this phase's reviewers):
- Sentence case always, plain verbs, no exclamation points.
- Unverified-login and expired-link states are routes to a calm screen, not error messages (D-05, D-06) — the routing itself IS the UX contract, not a copy afterthought.
- Password reset never reveals whether an email has an account (D-07) — the success copy above is the enforcement point; there is no "no account found" error path to accidentally build.
- Errors explain and guide, using structural weight (border, icon, semibold) via the existing `Input`/`Alert` `error` tier — never red, never blame ("you entered," "that doesn't," never "you failed to").

## Component Inventory (phase-specific addition to standard template)

Needs-driven only, per KIT-03's precedent — this phase composes the existing hardened kit, it does not add new base components. Auth screens are new **routes**, not new **kit components**.

| Screen / Route | Composition (existing kit only) | Notes |
|-----------------|----------------------------------|-------|
| `/signup` | `Card` > Fraunces `h2` heading + `Input` (name) + `Input` (email) + `Input` (password, notice hint) + `Button variant="primary"` + plain-text sibling link | D-03/D-04: 3 fields, one primary action, quiet link to `/login` |
| `/login` | `Card` > heading + `Input` (email) + `Input` (password) + `Button variant="primary"` + two plain-text links (signup, forgot-password) | Two sibling links is still zero competing *buttons* — both are `ghost`-weight text, not a second `Button variant="primary"` or `secondary"` |
| `/forgot-password` | `Card` > heading + body copy + `Input` (email) + `Button variant="primary"` → same Card swaps to the D-07 success copy in place after submit | Single field, single action; success state replaces the form in place, no navigation |
| `/reset-password` | `Card` > heading + `Input` (new password, notice hint) + `Button variant="primary"` | Single field per D-02/AUTH-04; arrived at already signed-in via `/auth/confirm` (D-08), so no email field needed |
| `/home` (interim, D-01) | `Card` > heading + body copy + `Button variant="primary"` (Log out) | The ONLY authenticated screen this phase; proves AUTH-05/AUTH-06 |
| check-inbox screen (path per Claude's discretion, e.g. `/check-inbox`) | `Card` > heading + body (email interpolated) + `Button variant="primary"` (Resend) + `Alert tone="notice"` shown after a successful resend | Shared by signup AND unverified-login per D-05 — one screen, one owner |
| expired-link screen (path per Claude's discretion, e.g. `/expired-link`) | `Card` > heading + body + `Input` (email, pre-filled when known) + `Button variant="primary"` (send fresh link) | Shared by verify AND reset expired/used links per D-06 |
| `/auth/confirm` (machine route, D-02) | No UI — Route Handler only, redirects to `/home` on success or the expired-link screen on failure (Pattern 2 in RESEARCH.md) | Not a visual screen; included here only so the inventory accounts for every route in D-02 |

All screens: centered single `Card` from the Phase 1 kit (D-03), `max-w-[440px]` matching the existing `/kit` page's column width, vertically centered in the viewport via `min-h-dvh flex items-center justify-center` (or equivalent) on the page wrapper. No new layout primitive — this is the same centered-column idiom already established, just centered vertically instead of scrolled from the top.

No new components are introduced. If implementation discovers a genuine gap (e.g., a shared `AuthCard` wrapper to avoid repeating the centering/max-width boilerplate across 7 routes), that is an acceptable **extraction**, not a new design element — it must compose `Card` unchanged and introduce no new visual language, per KIT-03's needs-driven rule.

Interaction/motion rules (inherited, KIT-05): existing two-tone focus ring, existing `prefers-reduced-motion` suppression, existing `Button loading` state (already built in Phase 1) — used here for real async submits (signup, login, password reset, resend) for the first time. No new motion is introduced.

Tap targets (inherited, KIT-04): every `Input` and `Button` in every screen above is the existing 56px floor — no exception.
