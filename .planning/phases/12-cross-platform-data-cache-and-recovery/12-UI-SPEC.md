# Phase 12 — UI Design Contract

> Visual and interaction contract for cross-platform data, cache, and recovery. Generated by gsd-ui-researcher; the checker upgrades `status` after verification.

---

## Phase UI Boundary

Phase 12 adds no reachable gallery screen, navigation destination, category control, settings screen, preview, export, source-message action, or deletion flow. Those surfaces belong to Phases 13 and 14.

Phase 12 must produce a provider-neutral presentation contract that lets those later surfaces render cache and recovery truth consistently on Android and iOS. The contract consists of:

1. one non-transient gallery-level notice state;
2. one retained-history boundary state;
3. one calm manual-retry action after the allowed automatic retry is exhausted;
4. distinct loading, authoritative-empty, cached, stale, incomplete, unavailable, and identity-ineligible states.

No cache-size control, data-use control, refresh control, or background-sync preference is permitted. Data Saver and Low Data Mode behavior is automatic.

Sources: `12-CONTEXT.md` D-07–D-18, `REQUIREMENTS.md` OFF-01/OFF-02, and the native direct-chat-only boundary in `AGENTS.md`.

## Spacing Scale

### Policy resolution: FISH tokens govern

The checker's generic reference scale of 4, 8, 16, 24, 32, 48, and 64 is not the governing scale for this repository. The precedence rule in `docs/ui-ux-agent-guidelines.md` puts FISH product rules first, the implemented design system second, resolved FISH guidance third, and generic advice last. That guide also resolves the grid conflict explicitly: use the named FISH spacing tokens, whose compact rhythm intentionally includes roles such as 12, 20, and 40, and never substitute an abstract grid for the implemented semantic scale.

Accordingly, Phase 12 introduces **zero new spacing values**. It must not rename, normalize, or replace an existing role with a generic-checker value such as 32 or 64. The table below is the allowed subset of existing generated platform roles, not a new phase-local scale:

| Existing token role | Generated value | Usage |
|---------------------|-----------------|-------|
| `twoXs` | 4dp / 4pt | Title-to-supporting-copy separation inside a notice |
| `xs` | 8dp / 8pt | Inline icon/action spacing |
| `sm` | 12dp / 12pt | Icon-to-copy gap |
| `md` | 16dp / 16pt | Notice internal padding |
| `page` | 20dp / 20pt | Horizontal gallery inset |
| `lg` | 24dp / 24pt | Notice-to-content separation |
| `xl` | 40dp / 40pt | Empty/unavailable state group separation |
| `twoXl` | 48dp / 48pt | Major state-region separation when a later screen needs it |

Exceptions:

- Interactive retry target: Android uses the existing 48dp `touchTarget`; iOS uses the existing 44pt `targetTouch`.
- Existing shared components may retain their generated 2dp/pt boundary and 12dp/pt control radius. Phase code must not restate those as local values.
- No one-off numeric spacing, phase-local aliases, or remapping to the generic checker scale is allowed. Layout code selects an existing platform token by semantic role.

Sources: `AGENTS.md` “Spacing discipline”; `docs/ui-ux-agent-guidelines.md` “Precedence,” “Token-only spacing,” and the resolved “Grid system” conflict; `design/tokens/fish.tokens.json`.

## Typography

This phase consumes exactly three existing Lexend semantic roles and two existing weights. It introduces neither a type scale nor a phase-local text style, and it does not add a screen heading or display role.

| Existing token role | Size | Weight | Line height | Phase 12 semantic use |
|---------------------|------|--------|-------------|-----------------------|
| `caption` | 13sp / 13pt | Regular 400 | 1.4 | Supporting notice copy and retained-history explanation |
| `label` | 14sp / 14pt | Medium 500 | 1.45 | Short notice title |
| `ui` | 15sp / 15pt | Regular 400 | 1.5 | Quiet `Try again` action |

The adjacent 13/14/15 values are intentional existing component roles, not interchangeable prose sizes:

- `caption` keeps multi-line explanatory copy subordinate without using muted color.
- `label` gives the notice title a medium-weight semantic step above its explanation.
- `ui` identifies an interactive action and preserves the established action-control metrics.

Consolidating these roles inside Phase 12 would require overriding the generated design system or restyling existing components, both of which are prohibited. Existing native code demonstrates the role split: iOS `Notice` uses `label` for its title and `caption` for supporting copy, while `ActionButton` uses `ui`; Android already exposes the same generated `caption`, `label`, and `ui` tokens, with `FishNotice` and `FishButton` retaining their component-owned role mappings. Phase 12 must not apply raw font sizes to force numerical parity. If a later Android notice extension adds separate title and supporting-copy slots, it must use the existing `label` and `caption` roles rather than inventing styles.

All copy uses Lexend, sentence case, Dynamic Type/font scaling, and unlimited vertical growth. Required state copy must not be truncated or forced to one line.

Sources: `design/tokens/fish.tokens.json`; Android `TypeTokens`, `FishNotice`, and `FishButton`; iOS `Typography`, `Notice`, and `ActionButton`.

## Color

Use semantic platform tokens rather than raw values.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | Android `FishTheme.colors.background`; iOS `Palette.bg` | Gallery canvas behind cached or live content |
| Secondary (30%) | Android `FishTheme.colors.surfaceAlt`; iOS `Palette.surface2` | The single gallery-level notice surface |
| Accent (10% maximum) | Android `FishTheme.colors.notice`; iOS `Palette.notice` | Informational icon and semantic notice boundary only |
| Text | `foreground` for title; `body` for explanation/action | Required state information remains readable and is never muted |
| Destructive | None | Phase 12 has no destructive user action |

Accent is reserved for the cache/recovery notice icon and its existing semantic boundary. It is not used for every cached item, thumbnails, category state, progress, or the retry action.

The retry action remains monochrome and quiet. No red/error treatment is allowed for offline, stale, incomplete, ordinary refresh failure, or purge-ineligible cache. Meaning must be expressed through icon, copy, placement, and semantics as well as color.

## Visual Hierarchy and Placement

### Gallery-level notice

- Render at most one notice for the entire gallery.
- Place it in the gallery content hierarchy above category content and below the eventual Phase 13 navigation/header region.
- The notice is inline and non-transient: it remains rendered while its state is true and never auto-dismisses like a toast.
- Use the existing notice surface, informational icon, title, and optional supporting copy. Do not add a badge or overlay to each item.
- Cached items retain their normal opacity, color, focusability, and layout beneath the notice.
- Do not replace useful cached content with a loading skeleton, blank screen, terminal error, or full-screen interruption during refresh or retry.

### Retained-history boundary

- Render once, immediately after the oldest retained cached item.
- Use plain text on the gallery canvas, not a card, badge, modal, toast, or per-item marker.
- Keep a larger gap before the boundary than between gallery items.
- The boundary communicates only that older content may exist. It must never say or imply that the local cache is the complete conversation history.
- Stale truth and incomplete-history truth are orthogonal: both the gallery notice and the history boundary may be visible at the same time.

### Geometry stability

- Refresh, automatic retry, and manual retry must not remove or reorder accepted cached items.
- Notice copy may update in place; the component footprint must grow vertically for larger text rather than clipping.
- Retry busy state preserves the action target and label footprint and prevents duplicate activation.
- Accepted server/realtime additions, replacements, and tombstones may update content only after the Phase 11 owner/request/cursor acceptance gates pass.

## Interaction Contract

### Recovery sequence

1. Gallery open, a meaningful foreground return, or connectivity return starts or joins one recovery cycle.
2. Coalesce overlapping lifecycle, reconnect, and realtime triggers; expose only one in-flight state.
3. Attempt refresh once.
4. If it fails while connectivity remains usable, retry automatically once after the deterministic short backoff. Do not expose a user choice during this automatic retry.
5. If the second attempt fails, retain cached content, mark it stale, and expose one quiet `Try again` action.
6. Manual retry starts one explicit new recovery cycle. It does not enable a background retry loop.
7. A successful accepted refresh removes stale/manual-retry truth and reconciles additions and tombstones without duplicates.

Defaults from `12-RESEARCH.md`: five-minute meaningful-foreground threshold, 500ms trigger coalescing, and one-second backoff plus deterministic 0–250ms jitter. These are testable shared constants, not visible settings.

### Manual retry

- Label: `Try again`.
- Treatment: ghost or underlined link-equivalent action inside the gallery-level notice; never the screen's primary action.
- Minimum target: 48×48dp Android; 44×44pt iOS.
- Enabled only when the second attempt failed and connectivity is currently usable.
- On activation, preserve geometry, prevent duplicate taps, and expose the busy state as `Trying again`.
- If connectivity is unavailable, show the offline cached or unavailable state without an inert retry control. Reconnect starts recovery automatically.
- On another failure, return to the same stale notice and retry action. Do not escalate color, copy, motion, or modal interruption.

### Data-saving behavior

- Android Data Saver and iOS Low Data Mode suppress one-screen lookahead only.
- Visible thumbnail loading remains allowed on a usable network.
- No banner, setting, prompt, or opt-in is introduced for data-saving mode.

### Identity transition

- The old owner's gallery content disappears before purge begins; never cross-fade or retain it beneath a loading state.
- While verified identity is missing, unresolved, or mismatched, prior cache is ineligible for display.
- No new-owner gallery content may appear until metadata, thumbnails, delivery references, decoded media, and temporary files for the old generation are purged.
- If cleanup cannot complete, fail closed: expose no old content. Other authenticated app surfaces may continue, but the gallery state is unavailable.

## State and Copy Matrix

Copy keys and state truth must match across Android and iOS. Platform punctuation and native accessibility phrasing may differ only when meaning stays identical.

| State | Content visibility | Notice / state copy | Action | Announcement |
|-------|--------------------|---------------------|--------|--------------|
| Initial online, no eligible cache | No gallery items yet | `Loading shared content` | None | Polite loading status once |
| Verified cache, refresh in progress | Keep cached items normally visible | `Checking for updates…` | None | Polite only if the refresh remains perceptible; do not announce every trigger |
| Offline with verified cache | Keep cached items normally visible | Title: `You're offline` Body: `This content is saved on this device and may be out of date.` | None | Polite once when state changes |
| Second refresh failure with cache and usable network | Keep cached items normally visible | Title: `Content may be out of date` Body: `We couldn't check for updates.` | `Try again` | Polite; focus does not move automatically |
| Partial retained cache, online | Keep retained items visible | Boundary: `More shared content may be available online.` | None in Phase 12 | Boundary is part of reading order, not a live alert |
| Partial retained cache, offline | Keep retained items visible | Boundary: `Connect to see more shared content.` | None | Boundary is part of reading order, not a live alert |
| Offline with no verified cache | No items | Heading: `Shared content isn't available offline` Body: `Connect to the internet to load it.` | None; reconnect is automatic | Announce once as the resulting state |
| Authoritative successful zero-item response | No items | Heading: `No shared content yet` Body: `Items shared in this conversation will appear here.` | None | Announce once after loading completes |
| Identity missing, unresolved, or mismatched | Prior-owner content hidden | No prior-cache copy or content may flash; remain in the authenticated shell's resolving state | None | Do not announce private prior-owner state |
| Purge/authority failure that makes cache ineligible | No items | Heading: `Shared content isn't available right now` Body: `Reconnect and try again.` | Retry only if the recovery contract says it can succeed | Polite once; never expose provider codes |

`Empty` is legal only after a successful authoritative response confirms zero eligible items. Offline/no-cache, identity-ineligible, permission-denied, and failed-loading states must never map to `empty`.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | None. Recovery is automatic and Phase 12 introduces no primary action. |
| Manual recovery action | `Try again` |
| Manual recovery busy state | `Trying again` |
| Cached notice | `You're offline` — `This content is saved on this device and may be out of date.` |
| Stale notice | `Content may be out of date` — `We couldn't check for updates.` |
| Online retained-history boundary | `More shared content may be available online.` |
| Offline retained-history boundary | `Connect to see more shared content.` |
| Empty state | `No shared content yet` — `Items shared in this conversation will appear here.` |
| Offline unavailable state | `Shared content isn't available offline` — `Connect to the internet to load it.` |
| Closed/unavailable fallback | `Shared content isn't available right now` — `Reconnect and try again.` |
| Destructive confirmation | None; Phase 12 has no destructive user action. |

Copy must never use `Error`, `Failed`, `Something went wrong`, `Your cache`, storage quotas, retry counts, provider names, status codes, signed URLs, file paths, or technical recovery language.

## Accessibility Contract

- Treat the notice title and body as one readable element; expose the retry action separately after it.
- Use an informational icon as a redundant cue and hide that decorative icon from TalkBack/VoiceOver.
- Announce meaningful state transitions politely. Coalesced triggers, one-screen lookahead, URL refresh, cache eviction, and the automatic retry are implementation details and must not produce repeated announcements.
- Do not move focus when refresh starts, cache becomes stale, or the notice copy changes.
- If the user activates retry, retain focus on the action through busy state and restore it to the same action if retry fails.
- If retry succeeds and the notice disappears, move focus only when required to avoid loss; otherwise preserve the currently focused gallery item.
- Cached, stale, incomplete, unavailable, and busy meaning must not rely on color.
- Support Dynamic Type/font scaling without truncation, overlap, or a horizontal scroll requirement.
- Preserve logical reading order: gallery context, one notice, category/content, retained-history boundary.
- Respect reduced motion. State changes use no required animation; if a later UI crossfades notice copy, use the existing 200ms fade and zero duration under reduced motion.

Phase 15 owns final TalkBack/VoiceOver, large-text, RTL, reduced-motion, contrast, and target-size release proof. Phase 12 must not encode state in a way that prevents those checks.

## Cross-Platform Parity Rules

- Android and iOS use the same semantic state names, copy keys, transition conditions, retry allowance, and action availability.
- Platform state may be presented through native Compose/SwiftUI components, but one platform may not show item badges while the other shows a gallery-level notice.
- `cached`, `stale`, and `retainedHistoryComplete` remain independently representable. A state can be cached, stale, and incomplete simultaneously.
- Visible-content preservation, retry count, reconnect reset, owner eligibility, and authoritative-empty rules must replay from the same TypeScript-owned fixture vectors.
- Presentation layers consume provider-neutral state. Supabase, Room, Core Data, signed URLs, filesystem paths, and provider error strings never enter display models.
- Delivery URLs, cache sizes, byte ceilings, eviction dates, or retry counters are never shown to the user.

---

## Prohibited UI

Phase 12 must not add:

- a gallery route or navigation entry;
- categories, tabs, filters, sorting, search, bulk selection, albums, or favorites;
- cache/data-use/offline settings;
- per-item stale, cached, unavailable, or download badges;
- a global synchronization banner, modal, toast, countdown, or progress percentage;
- a manual refresh button before the second recovery failure;
- a primary retry button or more than one recovery action;
- automatic full-content download, preview, export, source navigation, or deletion;
- any mobile home/dashboard or non-conversation surface.
