## Phase UI Scope

Phase 5 builds three user-facing surfaces:

| Surface | Contract |
|---------|----------|
| Client onboarding assessment | Warm in-chat assessment. A system voice asks one DB-configured question at a time; the client's selected or typed answer renders as their reply. No scroll form, no assessment picker, no plan/template choice. |
| Shared field renderer | Config-driven renderer/validator for `single_select`, `multi_select`, `scale`, `short_text`, `long_text`, and `boolean`. It must be reusable by Phase 6 tracker without onboarding-specific question IDs or copy branches. |
| Coach review | Read-only assigned-client review of submitted or partial onboarding answers, in assessment order, with the pinned version's prompt/options. No scoring, grading, editing, authoring, or plan assignment UI. |

Primary decision: preserve the `sketch-findings-fish` onboarding winner: conversational, chat-like, one question at a time. Do not copy sketch markup verbatim.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps, status glyph spacing, compact helper separation |
| sm | 8px | Bubble stack gaps, label-to-control spacing, answer chip internal icon gaps |
| md | 16px | Default component padding, message list side padding, grouped answer spacing |
| lg | 24px | Card padding, onboarding panel padding, coach review section padding |
| xl | 32px | Page/container gutters, major group separation, desktop onboarding column gap |
| 2xl | 48px | Empty/loading state vertical breathing room, major top/bottom page spacing |
| 3xl | 64px | Desktop page-level spacing only when the shell has room |

Exceptions:

- All interactive controls and answer chips must have `min-height: var(--size-control)` / 56px.
- Icon-only controls must be at least 56px by 56px with an accessible name.
- Progress track may use the existing 12px height from `Progress`; it is informational, not a tap target.
- Chat bubbles may use compact vertical padding, but any embedded answer action remains 56px tall.

Source: `AGENTS.md`, `docs/ui-ux-agent-guidelines.md`, `.claude/skills/sketch-findings-fish/references/theme-and-tokens.md`, `apps/web/app/globals.css`.

## Typography

Use exactly these four sizes in new onboarding UI. Do not introduce viewport-scaled type or additional weights.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Meta | 14px | 400 | 1.4 |
| Label / option | 15px | 600 | 1.4 |
| Body / control | 17px | 400 | 1.55 |
| Heading | 28px | 600 | 1.15 |

Font weights declared for this phase: regular 400 and semibold 600 only. Existing base components may retain their internal shipped styles, but new Phase 5 components must not add another weight.

Usage:

- System question prompt: 17px regular in a received chat bubble or system bubble.
- Answer chip label: 15px semibold when selected, 15px regular when unselected if the component supports state by weight.
- Autosave/status text: 14px regular, `text-muted` or `text-notice`.
- Page heading: 28px semibold Fraunces only on route-level context, not inside each question bubble.

Source: `docs/ui-ux-agent-guidelines.md`, `apps/web/app/globals.css`, `apps/web/components/ui/button/button.tsx`, `apps/web/components/chat/bubble/bubble.tsx`.

## Color

Use Tailwind utilities backed by `apps/web/app/globals.css`. Do not use raw hex values in web components.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `bg-bg` / `--color-bg` | Page canvas, chat background, coach review page background |
| Secondary (30%) | `bg-surface`, `bg-surface-2`, `border-border` | Received/system bubbles, cards, answer chips, progress track, review rows, skeletons |
| Accent (10%) | `bg-primary`, `text-on-primary`, `border-primary` | The single primary action per view; sent client answer bubble; selected-state emphasis where shape/weight also changes |
| Destructive | not used in Phase 5 | No destructive actions are in scope |

Accent reserved for:

- One `Button variant="primary"` per view when an explicit action is required.
- The user's saved answer bubble in the chat transcript.
- The filled portion of `Progress`.
- Selected answer chips only when paired with border/fill/weight shape changes; never color alone.

Feedback:

- Autosave, partial, offline, and validation messages use `notice` treatment first.
- Client-facing errors should prefer `Alert tone="notice"` or `Input notice` copy. Use `error` tone only for blocking invalid input that cannot be represented calmly by notice, and never with scolding copy.
- No red/alarming client-facing error slab.

Source: `AGENTS.md`, `.claude/skills/sketch-findings-fish/references/states.md`, `.claude/skills/sketch-findings-fish/references/theme-and-tokens.md`, `apps/web/app/globals.css`.

## Layout Contract

Client onboarding:

- Render inside the existing authenticated shell.
- Use a single focused chat thread/panel, maximum readable width 640px on desktop.
- On desktop, center the onboarding thread or place it as the active thread pane if the shell already uses a list/thread layout. Do not add a marketing hero or multi-card dashboard.
- On tablet and mobile, stay single-column. The question, current answer control, autosave status, and primary action must remain in one linear scan path.
- Use container-query-friendly component structure for any new responsive panel behavior. Avoid viewport-only layout assumptions for nested chat panels.

Coach review:

- Use the established coach client detail route pattern.
- Desktop may use a readable single column with grouped answer rows. Do not use a wide comparison table unless multiple clients are being compared, which is out of scope.
- Mobile stacks answer groups vertically with the prompt above the answer.

Source: `.claude/skills/sketch-findings-fish/references/responsive.md`, `.planning/phases/05-data-driven-onboarding/05-CONTEXT.md`, `apps/web/components/chat/chat-container/chat-container.tsx`.

## Interaction Contract

### Client Flow

- The client never chooses an assessment. The route loads the assigned/current active assessment version.
- Show one active question at a time.
- Keep a subtle bounded cue: `Question 2 of 7` is allowed. Do not show percentages, scores, grades, level predictions, or placement results.
- Show `Progress` visually with a quiet label, never a judgmental completion message.
- A selected/tapped answer becomes the client's reply bubble after persistence succeeds or enters a stable sending/saving state if optimistic UI is used.
- Refresh, route leave, and return must restore the last persisted draft and resume position from Supabase, not local-only state.
- Do not show "incomplete", "missed", "failed", "behind", or "required" as blame copy.

### Answer Types

| Type | Visual and interaction contract |
|------|---------------------------------|
| `single_select` | Large answer chips/radio-like buttons, one selected at a time. Selecting may autosave and advance if there is no competing primary button. Selected state uses border/fill/weight, not color alone. |
| `multi_select` | Large toggle chips with independent selected states. Requires one explicit primary action: `Save answer`. Include short helper text only if min/max selection rules exist. |
| `scale` | Bounded discrete steps as large segmented/chip buttons with plain labels from config. Do not use score-like copy, numeric grades, or percentages as judgment. |
| `short_text` | Use base `Input` with visible label derived from config. One explicit primary action: `Save answer`. Use `enterkeyhint="done"` or `"next"` as appropriate. |
| `long_text` | Use a textarea-style control matching `Input` dimensions/tokens: visible label, 56px minimum height, resize behavior that does not break layout, helper/error row reserved. One explicit primary action: `Save answer`. |
| `boolean` | Two large binary chips/buttons with calm labels from config, such as `Yes` / `Not right now`. May autosave and advance if no separate primary button is present. |

### Autosave and Status

- Draft persistence is visible but quiet.
- Use a stable 14px status row near the composer/control so status changes do not shift layout.
- Required statuses:
  - Saving: `Saving...`
  - Saved: `Saved`
  - Resume: `We saved your answers. You can continue when you are ready.`
  - Save failure: `That did not save yet. Keep this open and try again.`
  - Offline/connection: `You are offline. We will save when you are back.`
- The save failure state must preserve the typed draft and provide a focusable retry affordance if retry is possible.

### Completion

- Finalization uses one explicit primary action: `Share with coach`.
- Completion copy: `Your answers are ready for your coach. They will guide what comes next.`
- After completion, do not recommend a plan, level, score, or template.

Source: ONBD-01 through ONBD-07, D-01 through D-15 in `05-CONTEXT.md`, `.claude/skills/sketch-findings-fish/references/onboarding.md`.

## Component Inventory

Use existing components first:

| Component | Phase 5 use |
|-----------|-------------|
| `Button` | The single primary action, secondary back/retry where needed, loading state that preserves dimensions |
| `Input` | Short text answers; visible label, hint/notice/error row, 56px control height |
| `Card` | Coach review groups or genuinely framed tools only; no cards inside cards |
| `Progress` | Visual question progress; no visible percent-as-judgement |
| `Alert` | Calm notice/offline/save failure states |
| `ChatContainer` / `MessageList` / `Message` / `Bubble` | Conversational onboarding anatomy, adapted to assessment data |
| `Skeleton` | Loading question/answer history; no blank spinner-only screen |
| `EmptyState` | Calm no-answer / no-assessment states with no plan/template choice |

New Phase 5 components should be shared and config-driven:

- `FieldRenderer` or equivalent: accepts field config, current value, validation state, and callbacks; no hard-coded question IDs.
- `AnswerChip`: 56px minimum target, selected by shape/weight/border/fill, keyboard reachable.
- `ScaleField`, `BooleanField`, `TextAreaField`, and `MultiSelectField` if the renderer decomposition needs them.
- `AutosaveStatus`: stable-height 14px status row.
- `OnboardingQuestionBubble`: system/question bubble that can host the prompt and orientation cue.
- `CoachOnboardingReview`: read-only ordered prompt/answer groups.

Do not create a second button/input visual language for onboarding.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Client entry heading | `Let's get your coach a little context` |
| Client entry body | `Answer one question at a time. We save as you go.` |
| Primary CTA - start/resume | `Start onboarding` / `Continue onboarding` |
| Primary CTA - text or multi-select answer | `Save answer` |
| Primary CTA - finalization | `Share with coach` |
| Selection autosave status | `Saved` |
| Resume reassurance | `We saved your answers. You can continue when you are ready.` |
| Client empty/no active assessment | `Nothing to answer yet. Your coach will add the next step when it is ready.` |
| Client loading | `Loading your next question...` |
| Client save error | `That did not save yet. Keep this open and try again.` |
| Client config/error state | `This question needs a quick fix before you can answer it. Your coach can help.` |
| Completion heading | `Your answers are ready for your coach` |
| Completion body | `They will guide what comes next.` |
| Coach no answers heading | `No onboarding answers yet` |
| Coach no answers body | `When this client starts, their answers will appear here.` |
| Coach partial heading | `Answers are still in progress` |
| Coach partial body | `Review what has been saved so far. Nothing needs action yet.` |
| Coach denied/not found | `This client is not available here.` |
| Destructive confirmation | none; no destructive actions are in scope |

Copy rules:

- Sentence case only.
- Plain verbs, no technical codes as the only message.
- Never use `Submit`, `OK`, `incomplete`, `missed`, `failed`, `behind`, or scolding language.
- Do not say an assessment produced a level, grade, score, placement, plan, or recommendation.

## Accessibility Contract

- Every answer control is a real button, radio, checkbox, input, textarea, or semantically equivalent ARIA pattern with keyboard support.
- Use visible labels for text fields. Question prompt text may label a grouped control with `aria-labelledby`.
- Group single-select, multi-select, scale, and boolean answers with `fieldset`/`legend` or equivalent accessible group labeling.
- Every control target is at least 56px high.
- Focus must be visible on all answer chips, retry controls, and primary actions.
- Selected state must be perceivable without color: border/fill/weight/shape or an icon with text.
- Progress uses `role="progressbar"` with `aria-valuenow`, `aria-valuemin`, and `aria-valuemax`; visible copy may say `Question 2 of 7`.
- Autosave status uses a polite live region only for meaningful transitions, not noisy per-keystroke announcements.
- Error/notice text is connected with `aria-describedby`.
- Respect `prefers-reduced-motion`; skeleton pulse and message entrance must remain covered by existing global reduced-motion rules.

## States Contract

| State | Client surface | Coach surface |
|-------|----------------|---------------|
| Loading | Chat skeleton with next-question shape; no blank spinner-only screen | Skeleton rows matching prompt/answer groups |
| Empty | `Nothing to answer yet...`; no choices or CTA beyond normal navigation | `No onboarding answers yet`; no assignment/action button |
| Partial | Resume at last in-progress question with reassurance copy | Read-only partial banner: `Answers are still in progress` |
| Saving | Stable `Saving...` status row; control dimensions unchanged | Not applicable |
| Saved | Stable `Saved` status row, muted | Shows saved answer rows |
| Save failed | Notice tone, draft preserved, retry if possible | Not applicable |
| Submitted | Completion state with no score/recommendation | Read-only submitted answer list |
| Denied/not found | Calm unavailable state with no cross-client detail | `This client is not available here.` |
| Malformed config | Calm notice; do not render broken controls | Calm notice that the question cannot be shown |
