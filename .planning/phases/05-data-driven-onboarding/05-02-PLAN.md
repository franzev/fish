<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| DB config -> React renderer | Persisted JSON config crosses into UI rendering. |
| Browser input -> validation -> Server Action | Untrusted answer input is normalized before command writes. |
| Chat-style UI -> sensitive answer display | Personal answers are shown in bubbles and must not be overexposed. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Spoofing/Elevation | Renderer/action input contracts | mitigate | UI and schemas do not accept `client_id`; later actions derive user from session and command functions derive from `auth.uid()`. |
| T-05-02 | Elevation/Tampering | Service-role misuse | mitigate | This plan has no service-role code and no package installs; renderer works from passed data only. |
| T-05-03 | Information Disclosure | Field renderer and answer transcript | mitigate | Components render only current user's passed props; coach review is separate and RLS-scoped in later plans. |
| T-05-04 | Tampering | Malformed JSON config | mitigate | `fieldConfigSchema` and `validateFieldAnswer` reject malformed config/answer before render or save; DB CHECK is in Plan 05-01. |
| T-05-05 | CSRF/Tampering | Server Action payload shape | mitigate | `saveOnboardingAnswerSchema` defines exact POST payload accepted by later Server Actions. |
| T-05-06 | Tampering/Denial | Autosave race UI | mitigate | `AutosaveStatus` preserves draft status and Plan 05-03 discards stale saves; UI does not clear drafts on failure. |
| T-05-07 | Information Disclosure | Sensitive answer overexposure | mitigate | Components receive already-scoped data and tests assert no extra chooser/recommendation/result output. |
| T-05-SC | Tampering | npm/pip/cargo installs | mitigate | No package installs; reuse existing zod, React, Testing Library, and UI kit. |
</threat_model>
