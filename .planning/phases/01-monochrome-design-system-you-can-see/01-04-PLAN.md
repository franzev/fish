<objective>
Close the single blocking verification gap for Phase 1: the two-tone keyboard focus ring in `apps/web/app/globals.css` (lines 106-111) has its inner/outer `light-dark()` color pairs swapped, making the ring effectively invisible (~1.0:1 contrast, both bands) on the inverted primary Button in both light and dark themes — the single highest-priority control per screen (AGENTS.md "one primary action" rule), directly failing must-have truth #5 / requirement KIT-05.

The gap survived three human phase-gate approvals precisely because no automated test covered the rule. This plan fixes the swap AND installs the missing tripwire so it can never silently regress — following the exact pattern the phase already uses (parse `globals.css` at test time, assert with `colorjs.io` WCAG21, the same class of guard `theme-toggle.test.tsx` applies to the theme rules).

It also folds in one cheap, in-scope, zero-risk warning that strengthens this phase's own contrast contract: WR-04 (5 rendered token pairings absent from the contrast suite — all currently pass, unguarded against regression).

Purpose: A keyboard user can see where focus lands on the primary button in both themes — the accessibility floor this design system claims — provably and permanently.
Output: Corrected `:focus-visible` rule, a new `tests/focus-ring.test.ts` regression tripwire, and 5 added assertions in `tests/contrast.test.ts`.
</objective>

The defective rule as it currently ships (apps/web/app/globals.css lines 106-111):
  :focus-visible {
    outline: 2px solid light-dark(oklch(1 0 0), oklch(0.15 0 0));      /* WRONG — outer band */
    outline-offset: 2px;
    box-shadow: 0 0 0 4px light-dark(oklch(0.15 0 0), oklch(1 0 0));   /* WRONG — inner band */
    border-radius: 6px;
  }

The two-tone design intent (from D-05 and the rule's own code comment): the OUTER band (outline)
must contrast the PAGE the button sits on; the INNER band (box-shadow) must contrast the INVERTED
PRIMARY FILL the ring hugs. On the primary button the fill equals --color-primary and the page
equals --color-bg.

Resolved token values already in globals.css (do not change these — they are the fixtures the fix
is measured against):
  --color-bg:      light-dark(oklch(0.98 0 0), oklch(0.15 0 0))   (page canvas)
  --color-primary: light-dark(oklch(0.15 0 0), oklch(0.98 0 0))   (inverted primary fill)
  --color-surface: light-dark(oklch(1 0 0),   oklch(0.20 0 0))    (secondary/ghost/input backdrop)

Contrast helper already used project-wide (apps/web/tests/contrast.test.ts):
  import Color from "colorjs.io";
  Math.abs(new Color(fg).contrast(new Color(bg), "WCAG21"))   // WCAG 2.1 ratio

Single-source-of-truth test convention (contrast.test.ts + theme-toggle.test.tsx): tests
readFileSync globals.css and parse the live values — never hold a second copy of any color.
contrast.test.ts uses `// @vitest-environment node` because import.meta.url must be a file: URL;
the new focus-ring test must do the same (it reads globals.css and needs no DOM).
