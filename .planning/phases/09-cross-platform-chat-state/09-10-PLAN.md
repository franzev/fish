use-chat-realtime.ts today:
- Refs: `seenFirstSubscribeRef = useRef<Set<ReconnectChannelKey>>(new Set())`, `backfillInFlightRef = useRef<Promise<void>|null>(null)` — NEVER reset when `chat.conversationId` changes.
- Message-subscription effect (deps include `chat.conversationId`, `setRealtimeStatus`): `setRealtimeStatus(chat.conversationId, "connecting")` then `return subscribeToConversationMessages(id, onMessage, onConnected, onDisconnected)`. onConnected sets status "connected"; onDisconnected sets "disconnected". The returned cleanup only removes the channel — it never sets an inactive status.
- Read-subscription effect: `subscribeToConversationReadStates(id, (readState) => { dispatchChatEvent({ type: "mergeReadState", conversationId: id, readState }); mergeReadState(readState); }, onReconnected)` — dispatches the SAME merge twice (direct event + injected callback). `dispatchChatEvent` is also used by the message effect for `mergeRemoteMessage` (keep that usage).
- `mergeReadState` (injected from use-chat-read-state.ts) routes through the store `mergeReadState` action -> one `dispatchChatEvent`.

Store status type (chat-selectors.ts): `selectRealtimeStatusForConversation(state, id)` returns `RealtimeConnectionState` ("idle" | "connecting" | "connected" | "disconnected"), default "idle".

chat-client.tsx status derivation (render-time): `hasConnected` initializes from `realtimeStatus === "connected"`; `isReconnecting = realtimeStatus === "connecting" && hasConnected`. Resetting the store status to "idle" on cleanup makes a remount initialize `hasConnected` as false.

chat-client.test.tsx: has a hoisted `realtimeMock` capturing message/read handlers and channel/subscribe; reuse it to drive incoming payloads and connection status. `chatStore` and `resetChatStoreForTests` are imported at the top.
