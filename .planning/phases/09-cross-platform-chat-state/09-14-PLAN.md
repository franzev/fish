<objective>
Close CR-01: bind the browser chat cache to the verified auth identity so no
account can read another account's local chat state, and stop treating a failed
sign-out as success.

Today the module-singleton Zustand store is cleared only by the explicit logout
button, and `useLogout` ignores `signOut().ok` before clearing and navigating
(09-VERIFICATION, Review-Finding CR-01 — CONFIRMED BLOCKING, and the enabled
high-severity security gate). This plan:

- Adds a cache-partition fingerprint (`ensureChatStoreOwner`) that purges the
  store whenever the verified user changes — covering non-button account
  transitions on the same tab.
- Mounts a client `ChatIdentityGuard` that also purges on sign-out / cross-tab
  session change via the Supabase auth-state listener.
- Makes `useLogout` branch on `signOut().ok`: on failure it preserves state and
  surfaces calm retry guidance without navigating.

Purpose: browser-local information disclosure across account identity
transitions is the live security defect. Output: store fingerprint, guard
component + mount, logout hardening, and success/failure/non-button tests.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| account A session -> account B on same browser | The dangerous boundary: a soft SPA transition keeps the JS module + Zustand singleton alive across identities. |
| browser auth state -> local chat cache | The auth-state listener and server-verified userId are the only triggers allowed to purge the cache. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-14-01 | Information disclosure | module-singleton chat store across accounts | mitigate | ensureChatStoreOwner purges on verified-identity change; ChatIdentityGuard also purges on SIGNED_OUT / cross-identity auth events. |
| T-09-14-02 | Elevation of privilege | cache-owner fingerprint | mitigate | The fingerprint is a purge trigger only, never an authorization input; RLS/Edge Functions stay authoritative (D-05, D-08), enforced by the store authority-boundary test. |
| T-09-14-03 | Repudiation | failed sign-out treated as success | mitigate | useLogout preserves state and surfaces retry on ok:false instead of half-completing a navigation. |
| T-09-14-SC | Tampering | npm installs | accept | No package installs in this plan. |
</threat_model>
