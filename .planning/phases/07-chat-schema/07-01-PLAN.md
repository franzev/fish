<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Authenticated user -> `send_chat_message` RPC | Untrusted caller chooses conversation id, body, and idempotency key; DB derives sender identity and verifies membership. |
| Authenticated user -> RLS-protected chat reads | RLS decides conversation/message/read-state visibility. |
| Service-role seed/admin -> database | Seed and verifier setup bypass RLS; never used as authorization proof. |
| Phase 8 Edge Function -> DB RPC | Future Edge Function should verify JWT and call the database-owned command rather than duplicating message integrity. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-01 | Spoofing/Elevation | `send_chat_message` | mitigate | Derive `sender_id` from `auth.uid()` and `sender_role` from `profiles`; no sender parameters. |
| T-07-02 | Information Disclosure | conversation/message RLS | mitigate | `private.is_conversation_member` composes `is_coach_of`/`is_client_of`; verifier covers member, unassigned, and cross-client reads. |
| T-07-03 | Tampering | stored messages | mitigate | No authenticated update/delete grant or policy; verifier attempts mutation. |
| T-07-04 | Repudiation/Tampering | duplicate sends | mitigate | Required `client_request_id`; unique `(conversation_id, client_request_id)`; idempotent same-key return and conflicting-key rejection. |
| T-07-05 | Denial/Integrity | invalid bodies | mitigate | DB CHECK and RPC validation reject blank/oversized messages. |
| T-07-06 | Information Disclosure | read-state rows | mitigate | Own-row insert/update policies and member-scoped reads; verifier blocks updating the other participant's read state. |
| T-07-07 | Supply chain | dependencies | mitigate | No package installs or upgrades in this phase. |
| T-07-08 | RLS bypass by verifier | verification harness | mitigate | Service role used only for setup; assertions use real anon-key `signInAs` sessions. |
</threat_model>
