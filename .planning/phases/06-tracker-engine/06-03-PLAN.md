<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| untrusted caller → assign-tracker | Any authenticated user (or the service-role key) can reach the function; verify_jwt proves authenticity, but authorization (is this caller a coach of THIS client) is re-checked in-function on a caller-scoped client. |
| assign-tracker → tracker_assignments (service-role write) | RLS-bypass is deliberate and re-guarded by the preceding membership check; coach_id/version_id are server-derived. |
| coach → /coach/clients/[id] tracker timeline | RLS (private.is_coach_of) scopes rows; unassigned/cross-client returns zero rows → calm not-found, no leak. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-11 | Elevation of Privilege | a client calls assign-tracker to self-assign | mitigate | Membership check requires the caller be a coach of the target; a client fails is_coach_of → 403; `checkTrackerSelfAssignRejectedViaFunction` (live). |
| T-06-12 | EoP / Information Disclosure | a coach assigns to an unassigned client (cross-client) | mitigate | Caller-scoped is_coach_of re-check → 403; `checkTrackerCrossClientAssignRejectedViaFunction` (live, not SQL-only per Pitfall 2). |
| T-06-13 | Spoofing | caller sends a forged coach_id/version_id in the body | mitigate | Body carries only clientId; coach_id from getUser(), version_id from active config; grep gate on the source. |
