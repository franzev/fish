<objective>
Build the client-facing tracker surface: the `TrackerRepository` (interface + Supabase impl), the `getClientTrackerData()` server data helper, the `saveTrackerEntryAction` Server Action, the `/tracker` route, the `TrackerClientFlow` draft-preserving wrapper, the `TrackerEntryForm` (multi-field-per-period form composing the reused `FieldRenderer`), the net-new `MilestonePath` progress component, and the assigned home entry.

This mirrors Phase 5's `/onboarding` route + repository + Server Action shape almost exactly. The two genuinely net-new pieces are the multi-field entry form composition (per UI-SPEC A-01) and the `MilestonePath` step list — both fully specified by 06-UI-SPEC.md. The milestone step list is driven by the coach-authored `tracker_milestones` config + `get_tracker_milestone_steps()` shipped in Wave 1; this plan computes the done/now/up-next STATE from that config source, not from an entries count.

Purpose: A client can log their assigned tracker's entry with a preserved draft on failure and see calm milestone-journey progress whose labels + states come from coach-authored config and whose reassurance copy names the real assigned coach — no picker, no grade, no streak, no hard-coded name.
Output: A working `/tracker` route reachable from home, with green targeted tests and the full design line held.
</objective>
