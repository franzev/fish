## Phase UI Scope

Phase 6 builds four surfaces, reusing rather than reinventing:

| Surface | Contract | Requirement |
|---------|----------|-------------|
| Client tracker render | The client's single assigned tracker, rendered from a versioned config (fields + cadence), reusing `FieldRenderer` from Phase 5. Assigned, never chosen — no tracker list, no picker. | TRAK-01, TRAK-02, TRAK-04 |
| Entry save with draft preservation | The composer/entry draft is preserved on save failure or navigation away; calm non-scolding notice on failure; no layout shift; nothing lost on refresh. | TRAK-03 |
| Milestone-journey progress | Visual progress as a milestone path, reward-only. No grade, no adherence %, no streak, ever. | TRAK-05 |
| Coach read-only entry timeline | From the coach roster → `/coach/clients/[id]`, a read-only chronological timeline of the client's entries. No scoring UI, no editing, calm empty/partial states. | TRAK-06 |

Primary decision: this phase does NOT introduce a second visual language. The tracker entry
screen is the onboarding conversation's sibling — same field renderer, same autosave status row,
same calm-states vocabulary — applied to a recurring (not one-shot) form. The coach timeline is
the sibling of `CoachOnboardingReview` — same read-only Card-list pattern, chronological instead
of question-ordered.

## Spacing Scale

Declared values (must be multiples of 4) — identical to Phase 5, no new scale introduced:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps, milestone dot-to-connector spacing, status glyph spacing |
| sm | 8px | Milestone step internal gaps, label-to-control spacing, entry row internal icon gaps |
| md | 16px | Default component padding, tracker panel side padding, timeline row spacing |
| lg | 24px | Card padding, tracker panel padding, coach timeline section padding |
| xl | 32px | Page/container gutters, milestone path vertical rhythm, desktop column gap |
| 2xl | 48px | Empty/loading state vertical breathing room, major top/bottom page spacing |
| 3xl | 64px | Desktop page-level spacing only when the shell has room |

Exceptions:

- All interactive controls (field renderer chips/inputs, the entry save button, milestone
  focusable elements if any) must have `min-h-control` (56px), matching Phase 5.
- The milestone path's dots/connectors are **decorative structure, not tap targets** — no minimum
  size applies to the dot glyph itself, only to any button that sits on the same row (there is
  none in this phase; the milestone path is read-only, see Milestone-Journey Progress below).
- `Progress` track keeps its existing 12px (`h-3`) height — informational, not a tap target.
- The coach timeline's entry rows are read-only `Card`s, not buttons; no 56px floor applies to the
  row itself, only to any real focusable control inside it (there is none — pure read display).

## Typography

Reuse the exact four sizes and two weights Phase 5 declared. Do not introduce a fifth size,
a third weight, or viewport-scaled type.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Meta | 14px (`text-ui-sm`) | 400 | 1.4 |
| Label / option | 15px (`text-ui`) | 600 | 1.4 |
| Body / control | 17px (`text-copy`) | 400 | 1.55 |
| Heading | 28px (`text-heading-sm`→`text-display` per route level, Fraunces) | 600 | 1.15 |

Usage:

- Tracker field prompt: 17px regular, matches onboarding's question-prompt treatment (this is the
  same `FieldRenderer` — no new prompt component).
- Milestone step label ("now" step calm phrase, e.g. "A little more to go this week"): 15px
  semibold for the current step's phrase, 14px regular/muted for done/up-next step labels.
- Autosave/save-status text: 14px regular, `text-muted` default, `text-notice` for failure/offline
  — identical token usage to `AutosaveStatus` from Phase 5.
- Coach timeline entry date/cadence meta: 14px regular, `text-muted`.
- Coach timeline entry answer values: 17px regular (same body size as onboarding answer cards).
- Page heading ("Your tracker" / client display name on coach detail): 28px semibold Fraunces,
  route-level only — never inside each entry row.

## Color

Use Tailwind utilities backed by `apps/web/app/globals.css`. Do not use raw hex values in web
components. Structural UI stays chroma-0; the only sanctioned colors anywhere in the product are
the teal brand logo (not used on this phase's surfaces) and the calm, desaturated `notice` /
`error` / `warning` / `success` alert tones.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `bg-bg` | Page canvas behind the tracker panel, the coach timeline page background |
| Secondary (30%) | `bg-surface`, `bg-surface-2`, `border-border` | Tracker `Card` panel, entry field controls, milestone track/connector line, coach timeline entry rows, skeletons |
| Accent (10%) | `bg-primary`, `text-on-primary`, `border-primary` | The single `Save entry` primary action; the filled/done milestone dot; the current milestone's `Progress` fill; a selected field-renderer chip (shape+weight, never color alone) |
| Destructive | not used in Phase 6 | No destructive client actions are in scope (no delete-entry UI this phase) |

Accent reserved for:

- One `Button variant="primary"` per view — `Save entry` on the client tracker screen. Nothing
  else on that screen or the coach timeline is `variant="primary"`.
- The **done** milestone dot fill and the **current** milestone's progress bar fill.
- Selected field-renderer answer chips, always paired with a border/fill/weight change (per the
  Phase 5 `AnswerChip` contract) — never color as the only signal.

Feedback:

- Save-in-progress, save-failure, offline, and resume messages use `notice` tone first — same
  token (`text-notice`, `Alert tone="notice"`) as Phase 5's autosave states. Never a red slab.
- `error` tone (`Input`'s heavier `error` prop, `Alert tone="error"`) is reserved for blocking
  malformed-config states only (the tracker cannot render at all) — not for a transient save
  failure, which stays `notice` per TRAK-03's "calm non-scolding notice."
- The milestone path uses **zero** color-as-judgement: no red "behind," no green "ahead." Done vs.
  now vs. up-next is signaled by fill/ring/hollow shape only (see Milestone-Journey Progress).

## Layout Contract

### Client tracker render (Home surface, per navigation-and-shell.md: Progress tab does not ship
until this phase is coach-validated — the tracker itself lives on **Home** this phase)

- Render inside the existing authenticated shell, the same route family pattern as `/onboarding`.
- One focused panel: a `Card` containing the tracker's current-period entry form, maximum readable
  width `--container-content` (640px) on desktop, matching Phase 5's onboarding panel width.
- Directly above or below the entry `Card`, the milestone-journey progress block (its own section,
  not nested inside the entry `Card` — milestones are a distinct concern from "fill in today's
  fields").
- On desktop, center the tracker panel in the content column; do not add a dashboard grid, a
  marketing hero, or multiple competing cards.
- On tablet and mobile, stay single-column: milestone block, then entry form, then save action —
  one linear scan path, matching the "one linear scan path" rule from Phase 5.
- Container-query-friendly component structure; no viewport-only assumptions for the milestone
  block's reflow (desktop wider path → mobile narrower path, same vertical list).

### Coach read-only entry timeline (extends `/coach/clients/[id]`)

- Use the established coach client detail route pattern (same page, appended section — mirrors
  how `CoachOnboardingReview` was appended below the identity/goal/level `Card`).
- Desktop: a readable single column of chronological entry `Card`s, most recent first. Do not use
  a wide comparison table or a calendar/heatmap grid (a heatmap reads as a scoreboard).
- Mobile: the same single column, no layout change needed since it is already single-column.
- No primary button on this surface — it is 100% read-only, identical to `CoachOnboardingReview`'s
  "no primary button, no editing" precedent.

## Interaction Contract

### Client Flow (TRAK-01, TRAK-02, TRAK-04)

- The client never chooses a tracker. The route loads the single active assignment
  (coach/seed-assigned via the `assign-tracker` Edge Function, out of this document's scope —
  there is no assignment UI to design because none exists client-side).
- The tracker renders from its pinned config version using `FieldRenderer` verbatim — same
  `single_select` / `multi_select` / `scale` / `short_text` / `long_text` / `boolean` contracts
  Phase 5 already shipped. Do not fork a tracker-specific renderer or duplicate `AnswerChip`.
- Cadence (daily/weekly) determines what "the current entry" means, but the *rendering* contract
  is identical regardless of cadence: one form, current period, one save action. Do not add a
  cadence picker or a calendar-of-past-periods browser this phase — that is out of scope per
  TRAK-01's "not a hard-coded template" (config drives fields + cadence, not UI complexity).
- Each field behaves exactly per the Phase 5 Answer Types table (reproduced below for this
  phase's reference — no changes):

| Type | Visual and interaction contract |
|------|---------------------------------|
| `single_select` | Large answer chips, one selected at a time, autosave-and-advance-safe (no competing primary needed at the field level). Selected state by border/fill/weight. |
| `multi_select` | Large toggle chips, independent selected states, requires the explicit `Save entry` action. |
| `scale` | Bounded discrete steps as large chip buttons, plain config labels — never numeric-grade or percentage copy. |
| `short_text` | Base `Input`, visible label from config, part of the same `Save entry` action. |
| `long_text` | `TextAreaField` matching `Input` token dimensions, 56px minimum height, reserved feedback row. |
| `boolean` | Two large binary chips with calm config labels (e.g. `Yes` / `Not right now`). |

- Because a tracker entry is a **form with possibly multiple fields per period** (not strictly
  one-field-at-a-time like onboarding), the entry screen may show all of the current period's
  fields together inside one `Card`/`<form>`, with **one explicit primary action: `Save entry`**
  at the bottom. This is a deliberate, documented deviation from onboarding's one-question-at-a-
  time flow — assumption A-01 below explains why.

### Entry Save + Draft Preservation (TRAK-03)

- The save flow reuses the `AutosaveStatus` component and its exact status vocabulary and tone
  rules from Phase 5, applied to explicit-save (not per-keystroke autosave) since a tracker entry
  saves via the one `Save entry` action, not continuously:
  - Saving: `Saving...`
  - Saved: `Saved`
  - Save failure: `That did not save yet. Keep this open and try again.` (`notice` tone, never red)
  - Offline: `You are offline. We will save when you are back.`
- **Draft preservation is the hard requirement.** On save failure, on navigation away, and on
  refresh, the in-progress (unsaved) field values must still be present when the client returns —
  this is a stronger guarantee than Phase 5's per-answer autosave, because a tracker entry can
  hold several unsaved fields at once before the single `Save entry` tap. The composer/entry state
  must be restorable from a durable store (client-side local persistence and/or a Supabase draft
  row), not lost to an in-memory-only React state reset.
- No layout shift when the status row changes text — reuse the reserved-height status row pattern
  (`min-h-field-message`) from `Input`/`AutosaveStatus`, not a status element that appears/
  disappears and reflows neighbors.
- Retry affordance: when save fails, the `Save entry` button itself is the retry — do not add a
  second "Retry" button (that would be a second competing action; one-primary-action rule holds).
- Never say "you lost your progress," "incomplete," or "you didn't finish" — these are exactly the
  scolding patterns Phase 5's contract forbade and TRAK-03 extends the same guarantee to trackers.

### Milestone-Journey Progress (TRAK-05) — see dedicated section below for the full visual contract

- The milestone block is **read-only and non-interactive** this phase (no tap-to-expand, no
  drill-in) — it is a calm status display, not a navigation surface. If a future phase makes
  milestones tappable, that is a new UI-SPEC, not a retrofit here.
- The milestone block sits above/beside the entry form, never inside the same `<form>` — saving an
  entry does not "grade" or advance a milestone visually in a way that reads as a score-tick;
  milestone advancement is coach-paced ("James adds each step as you're ready" per the settled
  Profile & Progress finding), not an automatic per-entry increment.

### Coach Read-Only Entry Timeline (TRAK-06)

- Entries render in strict reverse-chronological order (most recent first), one `Card` per entry,
  mirroring `CoachOnboardingReview`'s Card-per-item pattern.
- Each entry `Card` shows: the entry's period/date meta (14px muted), then each field's
  label→answer pair rendered with the same `formatAnswer`-style logic `CoachOnboardingReview`
  already implements (reuse/extend that helper rather than writing a second formatter) — no
  numeric scoring, no color-coded "good/bad" entry treatment.
- No edit affordance, no delete affordance, no comment/reply-inline affordance on this surface —
  entry review is pure read. (Coaching action on tracker data, if ever needed, happens through
  chat via the existing "tap message → menu" authoring pattern, not through this timeline.)
- Partial/empty states use the identical Alert/Card-heading pattern `CoachOnboardingReview` uses
  for its `in_progress` and no-answers states (see States Contract below for exact copy).

## Milestone-Journey Progress — Concrete Visual Contract (TRAK-05)

This is the single highest-risk surface in the phase (top ADHD-audience abandonment trigger per
AGENTS.md rule 5 and the settled `profile-and-progress.md` finding). Be maximally concrete.

### What it MUST show

- A **vertical milestone path** (list of steps), each step in one of exactly three visual states:
  - **Done** — filled `--ink`-equivalent dot (`bg-primary` fill, using the same inversion token as
    the primary button, per the settled finding's `✓` filled dot), connector line to the next step
    solid.
  - **Now** (exactly one step, the current one) — a ring/outline dot (`border-2 border-primary`,
    unfilled center) with a short **visual bar** (reuse the existing `Progress` component, its
    `value` representing "how much of the current step is done" — never a client-facing number)
    plus **one calm phrase** describing the current step in plain language, e.g. "A little more to
    go this week."
  - **Up-next** — hollow, muted dot (`border-border`, no fill), muted label text (`text-muted`),
    connector line to it dashed or lighter-weight than the done-to-now connectors.
- Step labels come from the coach-authored milestone config/copy (plain language, e.g. "Log three
  days this week"), never auto-generated from a numeric threshold rendered as a fraction.
- A short reassurance line above or below the path attributable to the coach, matching the
  settled finding's voice: e.g. "James adds each step as you're ready." (Substitute the assigned
  coach's actual display name at render time; do not hard-code "James" in the shipped component —
  that name only appeared in the sketch source as illustrative coach copy.)
- The terminal "caught up" state (no up-next step exists yet) shows the `Progress`-tab-precedent
  calm copy pattern: a heading confirming completion plus a second clause promising the coach will
  add the next step — see States Contract below for the exact required copy.

### What it MUST NOT show (hard guardrails, verified by checker + a source grep gate)

- No percentage, fraction (`3 of 5`), score, rank, grade, or adherence rate anywhere on this
  surface or its surrounding page.
- No day-count streak integer of any kind, resettable or not — the schema itself stores entries,
  not a streak counter (TRAK-05's own text), so there is nothing to render as a streak even
  accidentally; do not compute one client-side from entry rows either.
- No red/alarming treatment for a missed period, a gap, or an incomplete current step. A gap in
  entries must never visually demote a "done" milestone back to "up-next," never shrink/dim a
  filled dot, and never insert a "missed" marker into the path.
- No chart-line icon, line graph, bar chart, or any icon/motif that reads as an analytics
  dashboard — the settled navigation finding explicitly rejects a chart-line icon for Progress
  ("reads as a grade — use a milestone path"); the same rule applies to this Home-surface milestone
  block since Progress itself has not shipped yet.
- No numeric x-axis/y-axis grid of any kind.
- No color-as-judgement (green = good, red = behind). Done/now/up-next differ by **shape and fill
  only** (filled dot / ring dot / hollow dot), consistent with the theme's "active state by shape,
  not color alone" accessibility rule.

### Returning after a gap

- If a client returns after missing one or more periods, the milestone path renders exactly as it
  would have if they had returned on time — same done dots, same current "now" step (the coach
  controls when a step advances, not an automatic cadence-tick), same reassurance copy. There is
  no "you missed 3 days" copy anywhere. This is the reward-only, non-decrementing guarantee from
  `profile-and-progress.md` applied concretely: **the render logic has no code path that can make
  the milestone path look worse than it did before the gap.**

## Component Inventory

Use existing components first — this phase should create very few new components:

| Component | Phase 6 use |
|-----------|-------------|
| `Card` | Tracker entry panel; coach timeline entry rows; milestone block container if a framed panel is used |
| `FieldRenderer` (from Phase 5) | Renders every tracker field from config, unchanged — no tracker-specific fork |
| `AnswerChip`, `TextAreaField` (from Phase 5) | Underlying field controls, reused via `FieldRenderer`, not re-implemented |
| `AutosaveStatus` (from Phase 5) | Save/saving/failure/offline status row for the entry form, reused verbatim (copy adjusted per the Copywriting Contract below where it must say "entry" instead of "answers") |
| `Button` | The single `Save entry` primary action; no other button on the client tracker screen |
| `Progress` | The current milestone step's visual bar fill; no percent-as-judgement label ever passed to it |
| `Alert` | Calm notice/offline/save-failure/malformed-config states; coach timeline partial-state banner |
| `Skeleton` | Loading state for the tracker entry panel and the coach timeline (reuse the chat `Skeleton` shimmer-row pattern, not a spinner) |
| `EmptyState` (pattern, not necessarily the literal chat component) | Calm "nothing to log yet" / "no entries yet" states with no CTA that implies a choice |

New Phase 6 components, kept minimal and config-driven:

- `MilestonePath` (or equivalent): accepts an ordered list of `{ id, label, state: "done" | "now" |
  "up-next", currentStepProgress?: number }` plus the assigned coach's display name for the
  reassurance line. No hard-coded step count, no tracker-specific IDs baked in — config-driven like
  `FieldRenderer`.
- `TrackerEntryForm` (or equivalent): accepts the pinned tracker config's field list, current
  period's draft values, validation state, and save callbacks; composes `FieldRenderer` per field
  plus one `AutosaveStatus` row plus one `Save entry` `Button`. Mirrors `OnboardingConversation`'s
  composition role but for a multi-field-per-period form instead of one-field-at-a-time.
- `CoachTrackerTimeline` (or equivalent): read-only ordered entry `Card` list, directly mirroring
  `CoachOnboardingReview`'s structure (same no-answers / in-progress-banner / answer-formatting
  pattern), reusing its `formatAnswer`-style logic against tracker field configs instead of
  onboarding question configs.

Do not create a second button/input/chip visual language for trackers. Do not create a second
"milestone" concept that diverges from the one described in this document (e.g. do not also build
a numeric percent-complete badge "for internal use" — if it exists in the component tree at all,
the checker will flag it as a leak risk).

## Accessibility Contract

- Every tracker field control is the exact same real button/radio/checkbox/input/textarea pattern
  `FieldRenderer` already ships with keyboard support — no new control semantics to invent.
- Group tracker fields with `fieldset`/`legend` exactly as `FieldRenderer`'s `OptionGroup` already
  does; do not bypass this when composing multiple fields into one entry form.
- Every control target is at least 56px high (`min-h-control`), matching the existing floor.
- Focus must be visible on all field controls, the `Save entry` button, and the milestone block
  contains no focusable elements this phase (it is non-interactive, so no focus-order concern —
  but if any future interactive element is added, it must clear the same `:focus-visible` ring).
- The milestone path's done/now/up-next states must be perceivable without color: dot fill/ring/
  hollow shape difference plus a text label per step (never shape alone with no label, since a
  screen-reader user needs the state conveyed in text — use `aria-current="step"` on the "now"
  item and a visually-hidden state word, e.g. "Done. Log three days this week." / "Now. A little
  more to go this week." / "Up next. ...", so the state is programmatically exposed, not just
  visual).
- `Progress`'s `role="progressbar"` / `aria-valuenow` / `aria-valuemin` / `aria-valuemax` contract
  carries over unchanged when used for the current milestone step's bar.
- Save-status text uses a polite live region (`role="status" aria-live="polite"`, matching
  `AutosaveStatus`'s existing implementation) only for meaningful transitions (saving→saved→
  failure), not per-keystroke noise.
- Coach timeline entries are read with normal document flow and heading structure (entry meta as
  a labelled group or `h3`/`h4` per entry, matching `CoachOnboardingReview`'s existing `h3` usage)
  — no `aria-hidden` on content a screen-reader coach needs to review.
- Respect `prefers-reduced-motion`; the milestone path introduces no new animation, and any
  loading skeleton reuses the existing globally-guarded pulse.

## States Contract

| State | Client tracker surface | Coach timeline surface |

## Assumptions

Decisions made without a blocking question, because this agent cannot ask the user and a genuine
gap remained after reading all upstream sources. Each is the calm, ND-appropriate default most
consistent with the shipped system.

- **A-01 (entry form shape — all fields at once vs. one-at-a-time):** Onboarding renders one
  question per screen; a tracker entry may legitimately have multiple fields for a single period
  (e.g. "mood," "practiced today," "notes" all logged together). I assumed the entry form shows
  all of the current period's fields in one `Card`/`<form>` with one `Save entry` action, rather
  than forcing a multi-screen one-field-at-a-time flow per entry. Rationale: TRAK-03 talks about
  "an entry" (singular unit saved together) and "the entry draft" (singular), which reads as one
  cohesive save, not N sequential saves; a repeating daily/weekly form that made a client click
  through 3-5 screens to log one day would add friction against the "remove choices, remove
  friction" product rule. If a future coach validation shows one-field-at-a-time is better for
  cadence trackers, that is a small `FieldRenderer`-composition change, not a new component.
- **A-02 (coach display name in milestone/reassurance copy):** The sketch source used the
  illustrative name "James" for the settled reassurance copy pattern. I assumed the shipped
  component must substitute the client's actual assigned coach display name at render time (data
  already available via the existing `coach_clients` relationship from Phase 4), never hard-code
  a name. This matches how `CoachOnboardingReview`/`/coach/clients/[id]` already use real,
  data-driven identity rather than placeholder names.
- **A-03 (milestone step source):** TRAK-05 says "visual progress ... as a milestone journey" but
  does not specify where step definitions/count come from. I assumed milestone steps are
  coach-authored config (mirroring the tracker's own versioned-config discipline from TRAK-01/
  TRAK-04), not derived automatically from entry counts — this is what makes "coach adds each step
  as you're ready" (the settled copy) literally true rather than aspirational copy over an
  auto-computed bar. The exact schema for milestone config is a planning/research concern, not a
  UI-SPEC concern; this document only constrains what must render and what must never render.
- **A-04 (Progress tab vs. Home placement):** Per `navigation-and-shell.md`, the Progress tab does
  not ship until a coach validates the tracker technique, and the 3-tab shell keeps the tracker on
  Home in the interim. I assumed Phase 6's client tracker screen (including its milestone block)
  is a route reachable from Home, not a new 4th tab, consistent with that explicit guardrail.
  Should validation land within this phase's timeframe and the roadmap decide to ship the Progress
  tab, the milestone block described here moves as-is — no new visual contract needed, only a
  navigation change.
- **A-05 (draft persistence mechanism):** TRAK-03 requires the draft survive failure/navigation/
  refresh. This document assumes (but does not mandate) that a durable mechanism — most likely a
  Supabase-backed in-progress entry row analogous to onboarding's `in_progress` attempt status —
  satisfies the requirement better than browser-only `localStorage`, since a client may switch
  devices; the exact mechanism is a planning/technical decision, but the **UI contract** (stable
  status row, no layout shift, calm failure copy, `Save entry` as its own retry) holds regardless
  of which mechanism the planner chooses.
- **A-06 (malformed-config tone: notice vs. error):** For the client-facing "config needs a fix"
  state, I used `error`-adjacent Input styling reasoning from Phase 5's own contract ("no
  destructive actions in scope... error tone only for blocking invalid input that cannot be
  represented calmly by notice") but kept the actual *copy* calm and coach-attributing rather than
  technical. This mirrors Phase 5's identical malformed-config precedent verbatim.
