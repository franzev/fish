<objective>
Bound the two unbounded message reads and add the three new read paths this phase needs, all in the
service/action layer, all as direct RLS-protected Supabase selects (never on the write-oriented
`chat-command` Edge Function). Today `SupabaseChatRepository.getAssignedConversation()` selects the
entire message history with no `LIMIT`, and the seeded general channel already holds ~928 messages
— this is the concrete performance fault CLOAD-01 targets.

Purpose: Give the client a bounded newest window on open (fast time-to-first-message), a
cursor-based "load earlier" page, a bounded reconnect backfill, AND a bounded newest-window read for
the reconnect reset fallback — the raw data capabilities Plan 03 wires into the store/hooks.
Output: A bounded keyset SSR query returning `hasMoreOlder`/`oldestCursor`; `loadOlderMessagesAction`,
`backfillMessagesAction`, and `loadNewestMessagesAction` (the bounded newest-window read the reconnect
reset-fallback needs — review HIGH 10-02); extended `core.test.ts` and `actions.test.ts`.

Migration determination (schema-push gate): the keyset query orders by `(conversation_id,
created_at, id)` in both scan directions, which the existing composite index
`messages_conversation_created_id_idx` (migration 0010) already covers — Postgres uses efficient
backward index scans for the `DESC, DESC` keyset. Therefore this plan adds NO file under
`supabase/migrations/` and needs NO `supabase db reset`/`migration up` schema-push task. If a future
`EXPLAIN` on the live query shows a sequential scan, a follow-up index migration + a [BLOCKING]
schema-push task would be required; that is out of scope here per RESEARCH (HIGH confidence the
index suffices).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser/client → PostgREST (reads) | Client-suppliable `limit` and `cursor` cross into a keyset query filter string. |
| Database → client | RLS policy `"members read messages"` (`private.is_conversation_member`) scopes every returned row regardless of the WHERE/keyset/LIMIT shape. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-03 | Denial of Service | loadOlder/backfill/loadNewest `limit` param | mitigate | Clamp page size with `Math.min(limit ?? 40, 100)`; never pass a raw client value to `.limit()`. Impact is self-scoped (RLS bounds rows to the caller's own conversation), so ceiling is sufficient. |
| T-10-04 | Information Disclosure | Forged/malformed cursor in the keyset filter | mitigate | zod `safeParse` validates cursor as `{ iso-datetime, uuid }` before it reaches the query; RLS enforcement is independent of the WHERE shape, so a forged cursor can only ever return rows already visible to the member. |
| T-10-05 | Elevation of Privilege | Read path bypassing the API boundary | mitigate | Reads stay direct RLS selects (Pitfall #9); no new `action` case on the write-oriented `chat-command` Edge Function, so no write surface is widened. |
| T-10-SC | Tampering | npm/pnpm installs | accept | Zero new packages installed this plan. |
</threat_model>
