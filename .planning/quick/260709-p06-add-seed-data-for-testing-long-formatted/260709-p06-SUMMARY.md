# Summary — MessageBody rich-text renderer + kit seed data

## What changed

- **`MessageBody`** (new: [message-body.tsx](../../../apps/web/components/chat/message-body/message-body.tsx)):
  a self-contained, line-by-line rich-text parser/renderer for chat message
  bodies — no npm dependency, never `dangerouslySetInnerHTML`. Supports bold
  (`**x**`), italic (`*x*` / `_x_`), inline code (`` `x` ``), fenced code
  blocks (with optional language label, no inline parsing inside), headings
  (`#`/`##`/`###`), blockquotes (`> `), bullet/numbered lists with nesting via
  indentation, links (`[text](url)`), and preserved line breaks (single `\n`
  → `<br>`, blank line → new paragraph). Props: `{ body, mine?, className? }`.
  - **Security (T-p06-01, T-p06-02):** builds React elements directly (React
    escapes text by default); link hrefs are sanitized to `http:`/`https:`/
    `mailto:` only — anything else (`javascript:`, `data:`, etc.) is
    neutralized and rendered as plain text with no href. Links use
    `target="_blank"` + `rel="noopener noreferrer"` (T-p06-03).
  - **Contrast on both bubble variants:** links use `text-current underline`
    (inherits the bubble's own text color); inline code / fenced code wells
    use `bg-surface-2` on received bubbles and `bg-on-primary/10` on `mine`
    bubbles (an opacity step off the existing `on-primary` token, not a new
    color) so the well stays legible against the inverted `bg-primary` fill.
    Headings/blockquotes inherit `text-current` and override the global
    `h1,h2,h3` base-layer rule (Fraunces serif + `text-foreground`) so they
    don't clash with the bubble's own type/color.
  - Exported from `components/chat` (barrel) and via `message-body/index.ts`.
  - 17 unit tests ([message-body.test.tsx](../../../apps/web/components/chat/message-body/message-body.test.tsx))
    cover every feature plus the two security cases (`javascript:` and `data:`
    URLs neutralized).
  - Stories ([message-body.stories.tsx](../../../apps/web/components/chat/message-body/message-body.stories.tsx))
    cover each feature individually, a mixed-format story, and both `mine`
    variants via a `Bubble` decorator.
- **Wiring** ([message.tsx](../../../apps/web/components/chat/message/message.tsx),
  [chat-client.tsx](<../../../apps/web/app/(authenticated)/chat/chat-client.tsx>)):
  both the kit `Message` component and the production `ChatClient` now render
  message bodies through `<MessageBody body={...} mine={mine} />` instead of
  raw text, so formatting renders identically in both places. No other
  behavior changed (reply rendering, deleted-message italic path, metadata all
  untouched).
- **Kit seed data** ([mock-data.ts](../../../apps/web/app/kit/chat/mock-data.ts)):
  added `shortMessages` (2), `longMessages` (16 — one message per
  MessageBody feature, a very-long plain-text message, a message mixing
  heading+bold+list+link, an edge case combining nested bullet AND nested
  numbered lists, and a message with one long unbroken URL with no break
  points), and `superLongMessages` (26, several long) plus three matching
  `ConversationSummary` entries. Existing `mockMessages`/`conversations`
  exports untouched.
- **Kit page** ([page.tsx](../../../apps/web/app/kit/chat/page.tsx)): "Full
  container" now stacks four labeled `ChatContainer` instances (original,
  short, long, super-long) in `h-chat-demo` cards for side-by-side comparison.
  All other kit sections untouched.

## Deviations from plan

### Auto-fixed issues

**1. [Rule 1 - Bug] Link URL regex broke on nested parens, leaving a dangling `)` as literal text**

- **Found during:** Task 1 (writing the `javascript:` neutralization test)
- **Issue:** the initial link regex used `\(([^)]+)\)` for the URL, which
  stops at the *first* `)` inside the URL. Any URL containing a function-call
  shape (`javascript:alert(1)`) or a paren in the path (e.g. a Wikipedia-style
  `Article_(disambiguation)` link) split the match, leaving a stray `)`
  rendered as plain text next to a still-linked/still-visible partial URL.
- **Fix:** the URL capture group now tolerates one level of balanced parens:
  `((?:[^()\s]|\([^()]*\))+)`.
- **Files modified:** `apps/web/components/chat/message-body/message-body.tsx`
- **Commit:** 868196c4

**2. [Rule 1 - Bug] `MessageList` grouped-corner test broke after Task 2's mandated DOM change**

- **Found during:** Task 2 (wiring `MessageBody` into `message.tsx`)
- **Issue:** `Message` used to render `{body}` as a direct text child of the
  `Bubble` div, so `screen.getByText(...)` returned the `Bubble` div itself
  (which carries the `rounded-*` corner classes the test asserts on). Once
  `MessageBody` wraps that text in its own block container, `getByText`
  returns the innermost `<p>` instead, one level deeper.
- **Fix:** updated `components/chat/chat.test.tsx`'s "renders consecutive
  received messages as a tight connected stack" test to reach the `Bubble`
  div via `.parentElement?.parentElement` from the matched text node (and one
  additional hop for the row-wrapper assertions), rather than asserting on the
  text node's element directly. No test intent changed — only how the DOM is
  traversed to the same element.
- **Files modified:** `apps/web/components/chat/chat.test.tsx`
- **Commit:** 7383fd6c

## Known/deferred issue — NOT fixed (file explicitly out of scope)

`apps/web/app/(authenticated)/chat/chat-client.test.tsx` has the identical
root cause as deviation #2 above — two tests assert
`screen.getByText("First grouped message")` / `screen.getByText("First
received message")` directly `.toHaveClass("rounded-*-chat-inner")`, and now
that text renders one level deeper (inside `MessageBody`'s wrapper), those
two assertions fail:

- `ChatClient > connects consecutive outgoing bubbles with small internal end corners`
- `ChatClient > connects consecutive received bubbles with small internal start corners`

This file had pre-existing uncommitted WIP changes at the start of this task
and was explicitly out of scope (orchestrator instructed: do not add, commit,
or touch it). It was **not modified**. Whoever finalizes that file's WIP
should apply the same fix pattern as deviation #2: reach the bubble-radius
div via `.parentElement?.parentElement` from the matched text, or query the
enclosing div by class/role instead of the text node directly. `pnpm
--filter @fish/web test --run` on the current tree shows exactly these 2
failures (426 passed, 2 failed) — no other collateral breakage from this
plan.
