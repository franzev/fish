# Quick Task 260711-en2 Plan

Reorganize the active Next.js web application around clear App Router, feature, component,
and provider boundaries, then decompose the largest modules and remove proven dead code.
This is a structural refactor: preserve all URLs, page output, user copy, styling, ARIA behavior,
Supabase/RLS assumptions, state transitions, public package exports, and callable action/component
contracts. Do not add product choices, learning behavior, dependencies, routes, or a
`tailwind.config.js`.

## Fixed implementation facts

- The installed runtime is Next.js `16.2.9`, React `19.2.7`, and the App Router. Keep
  `apps/web/proxy.ts` (the Next 16 convention); do not recreate `middleware.ts`. Pages and
  layouts remain Server Components unless interaction/browser APIs require a narrow
  `"use client"` boundary. Server Actions may live outside `app/` when their module retains
  `"use server"`.
- `apps/web/app/(authenticated)/chat/` has no `page.tsx`; it is reusable feature code used by
  `apps/web/app/(authenticated)/channels/[id]/page.tsx`. Move it out of the route tree rather
  than turning it into another route.
- `packages/core/src/chat-state/` remains the framework-neutral source of truth. Do not move
  React, Next.js, Zustand, Supabase, or browser concerns into it, and preserve the published
  `@fish/core/chat-state` API and fixture vectors.
- `apps/web/components/ui/` remains the shared visual primitive layer. Tailwind v4 tokens stay
  in `apps/web/app/globals.css`; all existing calm-focus, one-primary-action, 56px control,
  focus, reduced-motion, copy, and spacing-token rules remain unchanged.
- Existing internal callers rely on action names and result shapes from
  `app/(authenticated)/chat/actions.ts`, the `ChatClient` props contract, the
  `@/components/chat` barrel, `createSupabaseServices()`, and the raw-client compatibility
  helpers under `lib/supabase/`. Preserve those entry contracts through re-exports/adapters
  during the move; remove an adapter only when repository-wide search and tests prove it is
  no longer an active compatibility surface.
- The current Storybook glob only scans `components/**/*.stories.*`. If reusable stories move
  under `features/chat/components`, extend the glob to that exact location; do not scan all of
  `features/` and accidentally treat orchestration/server code as story candidates.

## Tasks

### Task 1 — Establish App Router and feature boundaries without changing APIs

Rehome the non-route chat module from `apps/web/app/(authenticated)/chat/` into
`apps/web/features/chat/` with explicit layers:

- `components/`: `chat-client.tsx` plus the current chat presentation folders from
  `apps/web/components/chat/`. Keep `ChatClient` as the client composition root and keep
  reusable primitives in `apps/web/components/ui/`.
- `hooks/`: the existing interaction/lifecycle hooks.
- `model/`: `chat-state.ts`, `message-grouping.ts`, `presence.ts`, and browser realtime
  primitives, with the Zustand adapter/selectors under `model/store/`.
- `server/`: the existing Server Actions and their server-only helpers.

Move colocated tests and stories with their implementations. Add narrow `index.ts` entry
points for client-safe feature exports, server actions, and store lifecycle/selectors; never
create one barrel that mixes client and server modules. Update the channel page, auth/logout
cache-reset callers, shell tests, and all source-scanning tests to import through those entry
points. Keep `apps/web/app/(authenticated)/channels/[id]/page.tsx` as a thin Server Component
that loads data, redirects, and supplies actions/data to the client boundary. Preserve
`@/components/chat` as a compatibility barrel that re-exports the same component names/types
from the feature, and replace the two existing deep Avatar imports with that barrel. Do not
rename routes, action functions, props, DTOs, Zustand events/selectors, or user-facing output.

Strengthen `apps/web/tests/service-boundary.test.ts` and add
`apps/web/tests/module-boundaries.test.ts` so future changes mechanically enforce: no
non-route feature directories under `app/`; no Supabase/provider imports from TSX; no
server-entry import from client modules; no app-route imports from reusable components/lib;
and no framework/provider dependencies in `packages/core/src/chat-state/`.

### Task 2 — Decompose high-complexity modules and consolidate imports

Refactor by extraction, keeping existing facade exports and behavior tests green after each
slice:

1. Split the roughly 1,160-line chat action module into server-only modules for schemas and
   public result types, response/DTO mapping and reaction/sender enrichment, Edge Function
   transport/error mapping, local-Supabase fallback commands, and the thin exported action
   entry points. Keep validation order, timeouts, local fallback rules, calm notices, access
   token handling, and every exported action signature byte-for-byte compatible at the public
   entry point.
2. Split the roughly 740-line `ChatClient` by extracting focused chat header/search,
   connection/load notices, message-list/message-row rendering, and composer/action surfaces.
   Keep state ownership and lifecycle hooks in the composition root unless moving them is
   required to eliminate prop threading; do not introduce effect-derived state, blanket
   memoization, new context, or duplicated stores. Preserve semantic HTML, keyboard behavior,
   focus restoration, scroll anchoring, optimistic/realtime reconciliation, and all tokenized
   classes.
3. Split `apps/web/lib/services/supabase/core.ts` by repository responsibility (auth,
   profiles/client profiles, coach-client assignments, chat, storage/realtime plus shared
   mapping/error helpers), leaving `createSupabaseServices()` as the stable factory that
   assembles the same immutable `SupabaseServices` shape. Split `apps/web/lib/auth/server.ts`
   into focused server data loaders while retaining its current named exports as a facade so
   pages and tests see no API change. Keep authorization in RLS/command boundaries and do not
   move data access into React components.

Within and after these moves, normalize cross-module imports to `@/...`, reserve relative
imports for files within the same component/module folder, use named component exports, and
keep `forwardRef` on focusable controls. Resolve duplicate DTO/helper logic only where the
same invariant is demonstrably identical; retain separate mappings when they represent
different trust/runtime boundaries.

### Task 3 — Prove cleanup, retain lean stories, and run every gate

Perform a repository-wide reachability/import audit after the new facades are in use. Remove
only source files, tests, exports, compatibility adapters, comments, and artifacts that have
zero remaining callers and no documented/public contract; delete tracked `.DS_Store` files.
Do not delete the raw Supabase client adapters merely because a preferred service API exists:
first migrate their auth/route/proxy callers safely or keep the small adapters as intentional
compatibility boundaries. Do not change generated database types or migrations as part of a
folder refactor.

Keep Storybook focused on reusable visual contracts. Move/update existing stories with their
components, add stories only for newly extracted reusable visual components that have
meaningfully distinct empty/loading/error/long-content or interaction states, reuse
`story-data.ts`, and avoid stories for pages, Server Actions, hooks, stores, repositories, or
thin composition wrappers. Update `.storybook/main.ts` narrowly if stories now live under
`features/chat/components`. Preserve all existing story exports and ensure no duplicate legacy
stories remain.

Run formatting/lint autofixes only on touched files, inspect `git diff --check`, then run all
gates from the repository root:

```bash
pnpm lint
pnpm typecheck
pnpm --filter @fish/web test -- --run
pnpm build-storybook
pnpm build
pnpm --filter @fish/web e2e
```

The Playwright command is part of completion, not optional; start/reset local Supabase and
provide `apps/web/.env.local` as required by the existing seed-based harness. Also run the
existing backend integration verifiers against that same local stack because the refactor
touches chat/provider boundaries:

```bash
pnpm verify:rls
pnpm verify:chat-realtime
```

Finish with repository searches proving there are no imports from the removed
`app/(authenticated)/chat` implementation path, no stale deep component imports that bypass
the compatibility/public entry points, no client-to-server imports, no unreferenced duplicate
implementation files, and no Storybook stories outside the intentional reusable-component
surface. Record any intentionally retained compatibility facade in a short module comment so
it is not mistaken for dead code later.

## Constraints

- Pure refactor: no product, schema, route, URL, copy, styling, interaction, auth, persistence,
  realtime, or public API changes.
- Use pnpm only. Add no dependencies unless the existing toolchain cannot perform a required
  check; no dependency is expected for this plan.
- Preserve user changes in the worktree and use `git mv` for relocations so history remains
  reviewable. Make small, testable extraction commits rather than one opaque rewrite.
- Do not weaken or delete a behavior/architecture test to make a move pass; update path-based
  assertions to the new intended boundary and retain the behavior assertion.
- Do not add speculative abstractions, generic `shared/` dumping grounds, default-exported
  components, index barrels that cross server/client boundaries, or Next.js route files that
  are not needed by current behavior.
