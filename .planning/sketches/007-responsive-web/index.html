Design question

Does “native per platform” hold as the web window resizes?

The same web app — the Messages screen — at three widths, driven by CSS container queries. Watch it collapse from a desktop two-pane, to a tablet rail + single pane, to a mobile bottom-nav layout that matches the Android idiom. One responsive system, native-feeling at every size.

Desktop — rail + two-pane

≥ 900px: the left rail (labeled), the conversation list, and the open thread all sit together — the classic desktop messaging layout (WhatsApp / Messenger Web).

fishhub.space/messages

Tablet — rail + single pane

600–899px: the rail stays, but there’s only room for one pane, so the open thread takes the space and a back control returns to the list. Nothing feels cramped.

fishhub.space/messages

Mobile — bottom nav, single pane

< 600px: the rail becomes a Material-style bottom navigation and you land on the conversation list — the exact mobile pattern the Android app uses. The web quietly becomes the app.

What to compare. It’s the same markup and tokens in all three — only the container width changes. The desktop two-pane and the mobile bottom-nav aren’t two designs; they’re one responsive system reflowing. That’s what makes web feel native at every size while staying consistent with Android. (Real build: use container queries so a panel reflows to its own width, not just the viewport’s.)
