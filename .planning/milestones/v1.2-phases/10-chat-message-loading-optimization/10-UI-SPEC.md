# Phase 10 — UI Design Contract

> Visual and interaction contract for bounded chat history loading, pagination, and reconnect states. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

## Design Intent

The canonical surface is the existing community thread at `/channels/general`. This phase does not redesign chat. It makes the current transcript open faster and load older history without shifting the message under the reader's eye.

The visible hierarchy remains:

1. The transcript is the task and stays fully readable.
2. Loading and connection feedback is quiet, inline, and temporary.
3. Send remains the only primary action.

The UAT correction is binding: while older history is loading, show only message-shaped skeleton placeholders. Do not show visible loading copy, a spinner, a generic full-width bar, or the `Load earlier messages` button at the same time.

## Spacing Scale

Only existing named spacing tokens may be used. The Phase 10 visible states use this subset:

| Token | Value | Usage |
|-------|-------|-------|
| `spacing-2xs` | 4px | Metadata-to-body separation and compact row padding |
| `spacing-xs` | 8px | Author/time and closely related placeholder gaps |
| `spacing-sm` | 12px | New-author message-row vertical padding |
| `spacing-md` | 16px | Avatar-to-content gap and transcript side padding |
| `spacing-page` | 20px | Existing small-screen page edge where inherited |
| `spacing-lg` | 24px | Larger separation only where the final message row already uses it |
| `spacing-xl` | 40px | Existing page rhythm only; not inside the loader |
| `spacing-2xl` | 48px | Existing large rhythm only; not a generic skeleton-bar height |

Semantic sizes:

| Token | Value | Contract |
|-------|-------|----------|
| `size-control` | 56px | Minimum height for `Load earlier messages` and `Try again` |
| `size-pagination-slot` | 104px | Reserved older-history feedback region; must not change between idle, loading, and error states |
| `spacing-badge` | 20px | Existing extra-small inline avatar/token only |

Exceptions: none. Existing 2px/6px tokens are not needed for the pagination-loading treatment. If the shared final message layout changes and 104px no longer contains the two-row skeleton exactly, update `--size-pagination-slot` as a semantic design token first; do not patch the component with a one-off height.

## Typography

Phase-owned feedback uses exactly four existing sizes and two weights:

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Meta/status | 12px (`text-ui-2xs`) | 400 | 1.35 |
| Timestamp/supporting label | 14px (`text-ui-sm`) | 400 | 1.45 |
| Message/loading body geometry | 17px (`text-copy`) | 400 | 1.55 |
| Compact section heading, if inherited | 20px (`text-heading-sm`) | 500 | 1.15 |

Only regular (400) and medium (500) are introduced or specified by this phase. Skeleton atoms have no visible text, but their boxes must inherit the same line box and layout geometry as the corresponding final author, timestamp, and message content. Do not use heading/display typography for loading feedback.

## Color

Use semantic tokens, not resolved hex/OKLCH values in components.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `bg-bg` | Transcript canvas and surrounding page |
| Secondary (30%) | `bg-surface`, `bg-surface-2`, `border-border` | Existing message surfaces, avatar wells, skeleton atoms, quiet dividers |
| Accent (10%) | `bg-primary` + `text-on-primary` | Existing own-message bubbles and the existing Send action only; Phase 10 adds no accent usage |
| Notice | `text-notice` / `Alert tone="notice"` | Offline and older-page-load failure guidance only |
| Destructive | Not used | This phase adds no destructive action or destructive state |

Accent is reserved for the existing Send action and existing authored-message treatment. Never apply accent to skeletons, `Load earlier messages`, retry, reconnect, offline, or pagination progress. Skeletons use `bg-surface-2` with an opacity-only pulse. Connection problems remain monochrome notice tone, never alarming red.

## Loading Skeleton Contract

### Required anatomy

The loading placeholder is a one-to-one structural representation of final community message rows, not two generic rounded rectangles. Render two deterministic rows inside the reserved 104px slot:

| Skeleton row | Must mirror | Required placeholder atoms |
|--------------|-------------|----------------------------|
| Row 1: new author group | The final `startsCommunityGroup` row using the same outer flex, `gap-md`, `px-md`, and `py-sm` geometry | 32px `Avatar size="sm"` circle; author line; timestamp line on the same metadata baseline; optional role-pill space only when the final shared row always reserves it; one message-content line |
| Row 2: same-author continuation | The final grouped continuation row using the same outer flex, avatar spacer, content alignment, `px-md`, and `py-2xs` geometry | 32px transparent avatar spacer; one message-content line aligned exactly with Row 1 content |

Every visible final-message element represented by the chosen skeleton rows must have a matching placeholder. The skeleton must reuse the final row wrappers or a shared layout primitive so avatar position, metadata baseline, content start, row padding, and total height cannot drift independently.

### Shape and motion

- Avatar placeholders are circular (`rounded-pill`); metadata/content placeholders follow the final element's radius, never one repeated generic radius for every atom.
- Use `bg-surface-2` for all visible skeleton atoms. No border, shimmer gradient, hue, icon, spinner, or visible copy.
- Apply the existing `animate-skeleton-pulse`: opacity 0.4 → 0.7 → 0.4 over 1400ms. Do not translate, scale, sweep, or stagger atoms.
- The global reduced-motion rule must reduce the pulse to one effectively static pass. The non-animated state must remain visibly identifiable as loading through shape and `aria-busy`, not motion alone.
- Skeleton geometry is deterministic across light/dark themes. Theme changes may alter token color only, never dimensions.

### Accessibility

- Set the message log or older-history region to `aria-busy="true"` while the fetch is in flight.
- Mark visual skeleton atoms `aria-hidden="true"` so screen readers do not encounter fake message content.
- Provide one non-visible status announcement, `Loading earlier messages`, through an `sr-only` status node or the existing log busy semantics. Do not announce each skeleton atom.
- Loading completion must not move keyboard focus. Newly prepended messages may be announced politely once; do not announce every inserted row as an urgent alert.

## Interaction Contract

### Load earlier

- `Load earlier messages` is a `Button variant="ghost"`, minimum 56px high, and never a second primary action.
- The IntersectionObserver sentinel and the manual button call the same `loadOlderAndPreserveScroll` callback.
- When a request begins, replace the button with the message-shaped skeleton block in the same reserved region. Do not retain the button label, show `Loading messages`, or invoke the Button spinner.
- Guard concurrent triggers: one request per conversation at a time. A still-visible sentinel must not create a retry storm.
- When no older history remains, remove the feedback region without inserting a celebratory, graded, or choice-heavy terminal state.

### Reading-position preservation

- Capture the viewport's `scrollHeight` and `scrollTop` before the prepend; restore by the height delta after the rows render.
- Restoration is instant. Never use smooth scrolling or an entrance animation for prepended history.
- Keep `.chat-log-viewport { overflow-anchor: none; }` so browser anchoring cannot double-compensate.
- The same message remains under the reader's eye after loading. The transcript below the reserved slot must not move when idle/loading/error states switch.
- Prepending history must not trigger the bottom-anchor behavior or display the `New messages` pill. Only a changed newest-message identity may do that.

### Failure and retry

- After a failed automatic load, stop automatic retries until the user explicitly retries or the conversation changes.
- Replace the skeleton with the existing notice-tone error and a quiet 56px `Try again` ghost button in the same reserved slot.
- Retry uses the same scroll-preserving callback as the sentinel and initial manual load.
- Keep focus on `Try again` through the retry transition where possible; do not jump focus into the transcript.

### Offline and reconnect

- Keep the transcript at full opacity; only the composer is de-emphasized while it cannot send.
- Offline state uses `Alert tone="notice"` and the exact calm copy below.
- `Reconnecting…` appears only after a conversation was previously connected. Ordinary first load must never be labeled reconnecting.
- Reconnect/backfill is automatic and must not surface a new action, modal, toast storm, or progress percentage.
- A reconnect reset to the bounded newest window must remain visually quiet, deduplicated, and correctly anchored; do not flash an empty transcript between windows.

## State Matrix

| State | Visible treatment | Interaction | Layout contract |
|-------|-------------------|-------------|-----------------|
| Older history available, idle | Centered `Load earlier messages` ghost button | Tap/click or sentinel loads | 104px reserved slot |
| Loading older history | Two-row message-shaped skeleton only | No duplicate trigger; log busy | Same 104px slot; transcript stays fixed |
| Older load failed | Notice Alert + `Try again` ghost button | Explicit retry | Same 104px slot; no automatic retry storm |
| No older history | No pagination affordance | None | No phantom button or completion grade |
| Initial connection | No reconnect copy | Subscription connects normally | Transcript/skeleton geometry unchanged |
| Genuine reconnect | Muted `Reconnecting…` | Automatic bounded backfill | Out of normal flow or space-reserved; no transcript jump |
| Offline | Notice banner: `You're offline. Reconnect, then try again.` | Past messages remain readable; failed send retries manually | No red; transcript remains full opacity |
| Gap exceeds bound | Quiet reset to newest bounded window | Automatic | No blank flash, duplicate, or smooth scroll |

## Responsive Contract

- Mobile, tablet, and desktop use the same message-row and skeleton anatomy; only the existing chat container width changes.
- The skeleton content column is fluid and follows the final message row's `min-w-0 flex-1` behavior. It must not overflow or introduce horizontal scrolling at 320px viewport width.
- The 32px avatar and 16px avatar-to-content gap remain aligned with final community rows at every breakpoint.
- `Load earlier messages` and `Try again` remain at least 56px high on coarse and fine pointers.
- Do not add breakpoint-specific skeleton counts, alternative loading copy, or a compact sub-56px control.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | `Send message` — existing single primary action; unchanged by this phase |
| Secondary pagination action | `Load earlier messages` |
| Non-visible loading status | `Loading earlier messages` |
| Empty state heading | None; keep the existing single-line empty state |
| Empty state body | Community: `No messages yet. Say hello to the community.` Direct thread fallback: `No messages yet.` |
| Older-load error | `Couldn't load earlier messages. Try again.` |
| Retry action | `Try again` |
| Offline state | `You're offline. Reconnect, then try again.` |
| Reconnecting state | `Reconnecting…` |
| Destructive confirmation | None — no destructive action is introduced |

All visible copy is sentence case, plain, and non-scolding. Do not add `Loading…`, `Please wait`, a message count, percentage, score, or technical network error. The skeleton is the only visible loading feedback for older history.

## Verification Contract

The implementation is visually acceptable only when all of the following are true:

- [ ] During an older-page request, the only visible loading UI is the two-row, message-shaped skeleton.
- [ ] Row 1 visibly contains avatar, author, timestamp, and message-content placeholders; Row 2 follows the exact continuation-row alignment.
- [ ] Skeleton and final rows share layout wrappers or a single extracted layout primitive; no duplicated approximation is accepted.
- [ ] The reserved feedback region remains 104px in idle, loading, and failed states, or the semantic token is deliberately updated to one new verified value.
- [ ] Loading three or more pages through both sentinel and button leaves the message under the reader's eye stationary and shows no duplicate messages.
- [ ] Prepending does not show `New messages` or jump to the newest message.
- [ ] Reduced-motion mode has no looping/flickering pulse and still communicates busy state semantically.
- [ ] Offline/reconnect feedback remains notice/muted, never red, and the transcript stays fully legible.
- [ ] The manual load and retry controls are at least 56px and preserve visible keyboard focus.
- [ ] No raw color, arbitrary Tailwind visual value, numeric spacing utility, extra primary action, new dependency, or registry block is introduced.
- [ ] Focused component tests assert skeleton anatomy, mutual exclusion with loading text/button/spinner, busy semantics, retry gating, and shared callback wiring.
- [ ] `pnpm lint`, `pnpm typecheck`, `pnpm build`, and focused chat tests pass.
