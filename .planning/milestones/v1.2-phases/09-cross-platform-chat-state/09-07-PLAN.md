<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Account A session -> account B session (same tab) | Soft logout/login does not reload the JS module, so the Zustand singleton persists across the boundary. |
| Browser UI -> Zustand store | Untrusted local coordination state (drafts, pending/failed sends) is cached here, keyed only by `conversationId`. |
| Store/hooks -> server actions/RLS | Durable auth, membership, and persistence remain on Supabase; the store is a cache only. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-71 | Information disclosure | Module-global `chatStore` across accounts | mitigate | `clearChatStore()` fully empties conversations + hydrationKeys on the soft-logout path; regression test asserts account B starts clean (CR-01). |
| T-09-72 | Tampering | Account B sending account A's leftover draft | mitigate | Composer draft and pending/failed local messages are cleared at sign-out, so no cross-account body can be submitted under account B. |
| T-09-73 | Elevation of privilege | Store treated as authorization truth | accept/mitigate | Store remains cache-only; RLS/Edge Functions still authorize every write. No auth/role/assignment fields added (boundary test enforced). |
| T-09-SC | Tampering | Package installs | accept | No package installs in this plan; no dependency graph changes. |
</threat_model>
