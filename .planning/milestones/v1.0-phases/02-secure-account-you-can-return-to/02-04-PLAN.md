<objective>
Ship the core linear auth loop: sign up (email + password + name, always a client) → verification email (FISH voice, token_hash link) → calm check-inbox screen → click link → `/auth/confirm` verifies and lands the person signed in at `/home` → one action, log out. Expired/used links route to a calm expired-link screen that resends the correct kind of email. An unauthenticated visit to `/home` redirects to `/login`. This slice, once green, is a user who can create an account and return to it — AUTH-01, AUTH-02, AUTH-05 (proven via the plan-01 proxy refresh), AUTH-06.

Purpose: The phase's headline capability, framed as the user story below. It composes the Phase 1 kit (Card + Input + Button + Alert) — no hand-rolled controls — and the plan-01 client factories.

Output: signup, check-inbox, expired-link, home routes; the `/auth/confirm` handler; a logout client island; the confirmation email template; and Vitest UI tests for every screen.

Review-driven changes in this revision (from 02-REVIEWS.md):
- HIGH: `/home` now explicitly redirects to `/login` when `getUser()` returns no user — the logout verification asserts this exact behavior, so AUTH-06 is meaningful instead of relying on "no longer shows an authenticated session" being undefined.
- MEDIUM: client screens reading `useSearchParams()` (check-inbox, expired-link) are wrapped in `<Suspense>` and the verify step runs `next build` so the App Router build gate (which errors on unbounded useSearchParams) passes.
- LOW: `/expired-link` takes a `type=signup|recovery` param so its single resend action calls the correct Supabase method instead of guessing.
- LOW: one-primary-action tests add an RTL `getAllByRole('button')` assertion alongside the source grep, so the check is not solely a fragile string count.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser form → Supabase Auth | Untrusted signup input crosses into GoTrue |
