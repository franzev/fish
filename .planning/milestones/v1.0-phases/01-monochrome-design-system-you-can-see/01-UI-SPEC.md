## Spacing Scale

Declared values (multiples of 4, matches existing Tailwind v4 default spacing scale — no custom `--spacing-*` tokens needed this phase):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-text gaps (Alert icon, notice/error icon spacing) |
| sm | 8px | Compact spacing — label-to-input gap, hint/notice text margin-top |
| md | 16px | Default element spacing — form field stacking, Card internal rhythm |
| lg | 24px | Section padding — Card padding, `/kit` section internal spacing |
| xl | 32px | Layout gaps — `/kit` section-to-section spacing |
| 2xl | 48px | Major section breaks — `/kit` page top-level rhythm |
| 3xl | 64px | Page-level spacing — `/kit` page vertical padding on desktop |

Exceptions:
- **Icon-only tap targets stay 56px** (`--size-control` token, KIT-04) even though the icon glyph itself renders at 20px — the 56px is the hit target, not the glyph size. This is not a spacing-scale exception, it's the existing tap-target floor applied consistently.
- No other exceptions. The existing codebase already uses Tailwind's default 4px-multiple spacing utilities (`px-4`, `px-6`, `mb-2`, `mt-2`, `p-6`, `gap-3`) — this phase does not introdu24 a new spacing token set, it continues the established pattern.

## Typography

Existing type scale (`apps/web/app/page.tsx`, `globals.css` base layer) already uses 3 sizes below the fold plus a display size in the hero — this phase formalizes it to exactly 4 declared sizes, 2 weights, consumed by the `/kit` demo page and Alert component.

| Role | Size | Weight | Line Height | Font |
|------|------|--------|-------------|------|
| Body | 17px | 400 (regular) | 1.55 | Lexend (`--font-sans`) — matches existing `body` base rule |
| Label / Caption | 14px | 400 (regular) / 500 (medium for field labels) | 1.4 | Lexend — form labels, hints, notices, captions |
| Heading (h3/section) | 20px | 600 (semibold) | 1.2 | Fraunces (`--font-serif`) |
| Display (h1/page title) | 32px | 600 (semibold) | 1.15 | Fraunces (`--font-serif`) |

Weights: **exactly 2** — 400 (regular, body/hints/notices) and 600 (semibold, headings + primary button label + error message emphasis per D-09). Where D-09 calls for error copy at "medium-weight," use 600 (the one semibold already in the system) rather than introducing a third weight (500) as a dedicated step — reserve 500 only for the existing field-label micro-emphasis already in `input.tsx` (`font-medium` on the `<label>`), which predates this phase and is not worth removing.

Line height: body 1.55 (existing, generous — fits ND reading-fluency goal), headings 1.15–1.2 (existing, tight and confident).

## Color

Pure monochrome, dual-theme, resolved via CSS `light-dark()` per D-16 — **not** hex values, since every token must resolve differently per color-scheme. All values are oklch, zero chroma (`0` in the chroma channel — TOKN-01), semantic-role-named only (TOKN-02, D-04). This table lists the full proposed token ladder (Claude's discretion per D-07), to be verified by the D-19 automated WCAG AA contrast test before lock-in.

| Role | Light | Dark | Usage |
|------|-------|------|-------|
| `bg` (60% dominant) | oklch(0.98 0 0) | oklch(0.15 0 0) | Page background — near-white / near-black, not true #fff/#000 (D-02) |
| `surface` (30% secondary) | oklch(1 0 0) | oklch(0.20 0 0) | Card, Input, elevated containers |
| `surface-2` (30% secondary, alt) | oklch(0.94 0 0) | oklch(0.26 0 0) | Alt rows, subtle wells, Progress track |
| `border` | oklch(0.85 0 0) | oklch(0.34 0 0) | Hairline separators, default field border |
| `border-strong` (new, KIT-02) | oklch(0.75 0 0) | oklch(0.44 0 0) | Notice-tier field/Alert border (D-08 "thicker/darker border") |
| `foreground` | oklch(0.15 0 0) | oklch(0.97 0 0) | Headings, highest-emphasis text |
| `body` | oklch(0.32 0 0) | oklch(0.88 0 0) | Body copy — verified ≥4.5:1 against `bg` and `surface` in both themes |
| `muted` | oklch(0.50 0 0) | oklch(0.68 0 0) | Captions, placeholders, hints, `Progress` label |
| `primary` (10% emphasis — full-contrast inversion, D-03) | oklch(0.15 0 0) | oklch(0.98 0 0) | The ONE primary action fill per screen — button only |
| `on-primary` | oklch(0.98 0 0) | oklch(0.15 0 0) | Text/icon on the primary fill |
| `notice` (icon + text tint, tier 1) | oklch(0.40 0 0) | oklch(0.80 0 0) | Notice-tier Alert/Input icon + message text — regular weight, `border-strong` border, no hue |
| `error` (icon + text tint, tier 2) | oklch(0.20 0 0) | oklch(0.95 0 0) | Error-tier Alert/Input icon + message text — semibold weight (600), `border-strong`-or-heavier border, no hue — this is the ONE "second semantic color" the product allows, and it is monochrome, not red |
| `success` | oklch(0.40 0 0) | oklch(0.80 0 0) | Success Alert/inline confirmation — same value as `notice` (structurally identical per D-11: "same structural language as notices with a different icon") |
| Focus ring inner | oklch(1 0 0) | oklch(0.15 0 0) | Two-tone ring, D-05 — inner light |
| Focus ring outer | oklch(0.15 0 0) | oklch(1 0 0) | Two-tone ring, D-05 — outer dark, guaranteed visible against inverted `primary` fill (Pitfall 4) |

**60/30/10 split, restated for a monochrome system without hue:** dominant = `bg` (the page canvas, ~60% of any screen); secondary = `surface`/`surface-2`/`border` (cards, inputs, dividers, ~30%); emphasis = `primary`, reserved **exclusively** for the single primary-action button fill on a screen (never for links, never for icons, never for decorative emphasis) — this is the "10%" role, expressed as contrast-inversion rather than hue, per D-03.

Emphasis (`primary`) reserved for:
- `Button variant="primary"` fill only (at most one per screen, per AGENTS.md rule 1)
- Nothing else. Not links, not active nav state, not icons, not focus rings (focus rings use the separate two-tone ring tokens above, not `primary`).

Second semantic color (destructive/error): `error` token above. Per AGENTS.md rule 6 and D-09, this is **not red** — it is the same neutral chroma-0 ladder, distinguished from `notice` purely by border weight (`border-strong` → heavier still) and copy weight (400 → 600), never by hue. There is no separate "destructive" red token in this system; destructive-action confirmation (Phase 2/3 scope) will reuse the `error` tone, not introduce a new hue.

Elevation (D-06): light theme uses soft `box-shadow` on `Card`/elevated surfaces (not yet in current CSS — add as `--shadow-card: 0 1px 2px oklch(0.15 0 0 / 0.06), 0 4px 12px oklch(0.15 0 0 / 0.06)` or equivalent, light-theme-only via `light-dark()` boxing to `none` in dark). Dark theme uses `surface`/`surface-2` lightness-step instead of shadow (already captured in the ladder above).

**Contrast verification is mandatory, not advisory:** every foreground/background pairing above must pass the D-19 `colorjs.io` WCAG21 AA script (4.5:1 text, 3:1 UI components) in both themes before these exact numbers are treated as final. The values in this table are the starting proposal (per RESEARCH.md Open Question 1); the `/kit` page is where they get visually judged and, if the script fails a pairing, adjusted — this is expected iteration, not a spec violation.

**Existing tokens being replaced (not additive):** `apps/web/app/globals.css` currently defines an aquatic-blue palette with `--color-old-hue-token-a` and `--color-old-hue-token-b` (hue-named, violates TOKN-01/TOKN-02) and a `--color-primary-press` state. This entire `@theme` color block is replaced wholesale by the monochrome ladder above — no hue tokens carry forward. `primary-press` (pressed/hover state for the primary button) becomes a lightness shift within the neutral ladder (Claude's discretion — e.g., light theme press = oklch(0.25 0 0), dark theme press = oklch(0.90 0 0); verify contrast + visually confirm on `/kit`).

## Copywriting Contract

This phase has no product screens with real user-facing copy (no auth, no empty states — those are Phase 2/3). The copywriting contract here covers the **demo page** and **component default/example copy** that KIT-06 requires to be shown, plus the tone rules downstream phases must inherit.

| Element | Copy |
|---------|------|
| Primary CTA (demo page example, not a real product action) | "Get started" (existing `page.tsx` example — carries forward into `/kit`, still calm/plain-verb) |
| Empty state heading | Not applicable this phase — no empty states exist yet (Phase 3 scope, SHEL-02). Tone rule to carry forward: never "No data found" — always a guiding, calm sentence. |
| Empty state body | Not applicable this phase — deferred to Phase 3. Tone rule: state what's true + what happens next, never blame the user. |
| Notice-tier message (Input/Alert example copy) | "That doesn't look like an email yet. Check the spelling?" (existing `page.tsx` example — plain verb, sentence case, question invites correction rather than declaring failure) |
| Error-tier message (Alert example copy, new) | "Something needs your attention before you can continue." (calm, structural — describes what's true, points at the fix; never "Error:", never "Invalid", never exclamation points) |
| Success message (Alert example copy, new) | "Saved." / "You're all set." (quiet, unceremonious per D-11 — no celebration, no exclamation mark, consistent with reward-only tone) |
| Destructive confirmation | Not applicable this phase — no destructive actions exist in the design-system/demo scope (no delete, no logout). Tone rule for future phases: name the action plainly ("Log out?" not "Are you sure?"), confirm with a plain verb button, never scare language. |

Copy rules inherited from AGENTS.md (binding on every phase, restated here since KIT-02 depends on them):
- Sentence case always, plain verbs, never scolding.
- Errors explain and guide; they use structural weight (border, icon, semibold), never alarm color.
- No exclamation points as a substitute for calm clarity.

## Component Inventory (phase-specific addition to standard template)

Needs-driven only, per KIT-03 — no speculative components.

| Component | Status | States required (KIT-01) |
|-----------|--------|---------------------------|
| `Button` | Harden existing (`apps/web/components/ui/button.tsx`) | default, hover, focus (two-tone ring), disabled, loading (new — needs a loading state per KIT-01; spinner or pulse respecting reduced-motion), for all 3 variants (primary/secondary/ghost), both themes |
| `Input` | Harden existing (`apps/web/components/ui/input.tsx`) | default, hover, focus, disabled (new), error (new — distinct from existing `notice` prop per D-08/D-09 two-tier split), notice (existing, rename semantics to match tier system), both themes |
| `Card` | Harden existing (`apps/web/components/ui/card.tsx`) | default, elevated (shadow in light / surface-2 step in dark per D-06), both themes |
| `Progress` | Harden existing (`apps/web/components/ui/card.tsx`) | default (0–100 value-driven fill), both themes — no numeric label as a grade (AGENTS.md rule 4 — visual only) |
| `Alert` | New (`apps/web/components/ui/alert.tsx`) | notice tier (IconInfoCircle, regular weight, `border-strong`), error tier (IconAlertCircle, semibold message, heavier border), success tier (IconCircleCheck, regular weight) — all 3 tiers required on `/kit` per Pitfall 5, both themes |
| `KitThemeToggle` | New, dev-only (`apps/web/components/kit/theme-toggle.tsx`) | system/light/dark override buttons, sets `color-scheme` on `html`; not part of the product-facing kit, not shown as a "component" section on `/kit` — it's the page's own control |

Icon usage (KIT-02, D-08/D-09/D-11): `IconInfoCircle` (notice), `IconAlertCircle` (error), `IconCircleCheck` (success) from `@tabler/icons-react`, size 20, stroke 1.75 (notice/success) or 2 (error — slightly heavier stroke reinforces tier without color), `aria-hidden="true"` since adjacent copy already carries the meaning.

Interaction/motion rules (KIT-05): two-tone focus ring (D-05) globally via `:focus-visible` in `@layer base` — inner light + outer dark ring, verified visible against both the inverted `primary` button fill and default surfaces in both themes (Pitfall 4). All transitions/animations wrapped by the existing `prefers-reduced-motion` suppression in `globals.css` (already present, must be preserved verbatim).

Tap targets (KIT-04): every interactive element — `Button`, `Input`, and any focusable element inside `Alert` (e.g., a dismiss control, if added) — enforces `min-h-[var(--size-control)]` (56px), the existing token, unchanged.

Demo page structure (KIT-06, D-12/D-13/D-15): single dedicated route `apps/web/app/kit/page.tsx`, one long sectioned scroll — tokens (color swatches, both themes visible via toggle), typography (Lexend/Fraunces samples at all 4 declared sizes), icons (the 3 Tabler icons in use), then component sections in this order: Button (all variants × all states), Input (all states × both tiers), Card + Progress, Alert (all 3 tiers). No nav, no route params, ships in production unlinked (D-15).
